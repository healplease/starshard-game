### 2. FAIL-loop proof — three QA-planted defects (independent of the Programmer's)

structurally can't see input-path bugs. I planted three defects — **distinct from the Programmer's three**
(theirs: flush no-op / boss-bar overlap / oversized label) — one per gate category, reverting after each.
**In every case the smoke gate stayed green (exit 0)** while the targeted gate went red.

| # | Planted defect (site) | Smoke | Gate that caught it (RED) | AC# / check |
|---|------------------------|-------|---------------------------|-------------|
| **D1** | `systems/bombs.py _flush` — drop `world.enemies.clear()` (behavioral / input-path) | **exit 0 (green)** | event-script gate (`bomb` FAIL) **+** regression harness **5** red | **AC30, AC32, AC40, AC50, event** |
| **D2** | `config.HP_BAR` → `(60,…)` so the HP bar overlaps the SCORE readout (a different rect pair than the Programmer's boss-bar) | **exit 0 (green)** | regression render-rect gate | **AC47** (HP bar vs score) |
| **D3** | `config.CONTROLS_1` padded past the 600 px panel (a different string than the Programmer's boss label) | **exit 0 (green)** | regression width gate | **AC47w** (string_widths) |

- **D1 is the crown (T1).** The smoke run *does* execute a scripted X-bomb @f20, yet because smoke only
  asserts `exit 0` it never notices the flush silently leaves the enemies alive. The behavioral
  **event-script** gate (which drives the real `_handle_events`) and the durable harness both go red. This is
  exactly the retro finding that "exit 0 can't see input-path bugs" — now demonstrated, not asserted.
- After each defect was reverted, the **full gate set was re-run green** (49/49, event 5/5, smoke exit 0,
  probe 10/10) and a `grep` confirmed **no `PLANTED DEFECT` residue** remains in `workspace/game/`.

deleted per `test_plan §1.5`), separate from the Programmer's seeds: it (a) asserts the run clock **freezes →
