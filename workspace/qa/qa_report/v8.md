## 3. v8 AC checklist (R69–R75)

### AC53 — Esc toggle PLAY↔PAUSE; no entity drift (R69 / R70)

| Check | Observable | Result |
|-------|-----------|--------|
| AC53a | PLAY + K_ESCAPE KEYDOWN → `app.state is GameState.PAUSE` | **PASS** |
| AC53b | PAUSE + K_ESCAPE KEYDOWN → `app.state is GameState.PLAY` | **PASS** |
| AC53c | Player `(x, y)` unchanged across a PAUSE cycle | **PASS** |
| AC53d | `world.score` and `world.frame` unchanged during PAUSE | **PASS** |

Freeze mechanism confirmed architecturally: `run()` only calls `_step_play(inp)` in the `elif
self.state is GameState.PLAY` branch. PAUSE uses a separate branch that reads the Q key only.
All 8 R70 freeze-list systems (`physics.update_play`, `encounter.update`, `bombs.update`,
`combat.resolve`, `spawning.update`, `buffs.tick`, `w.frame += 1`, `scoring.survival_tick`)
are all inside `_step_play()` and are therefore frozen by architecture — no individual guards
needed (GDD §V8.5 confirmed).

### AC54 — Boss run clock frozen during PAUSE (R70 bullet 3)

| Check | Observable | Result |
|-------|-----------|--------|
| AC54a | `run()` has `elif self.state is GameState.PLAY` guard (separate PAUSE branch) | **PASS** |
| AC54b | `w.frame += 1` is inside `_step_play()` only | **PASS** |

`w.frame` is the run clock used for boss breakpoint evaluation (`t = w.frame / 60.0`). Since
`_step_play` is not called in PAUSE, the clock does not advance and boss breakpoints cannot
be delayed or triggered by pausing. Confirmed by source inspection.

### AC57 — Q released before t_quit → arc resets; no accumulation (R72)

| Check | Observable | Result |
|-------|-----------|--------|
| AC57a | `run()` has `else: self.q_hold_frames = 0` when Q not held | **PASS** |
| AC57b | Simulated release: `q_hold_frames` resets to 0 immediately | **PASS** |

The `else` branch in the PAUSE Q-hold section unconditionally resets `q_hold_frames = 0` when
`pygame.key.get_pressed()[pygame.K_q]` is False. No accumulation across interrupted holds.
New press restarts from zero on the very next held frame.

### AC58 — Esc no-op in START/GAME_OVER; K_ESCAPE→quit removed; pygame.QUIT intact (R73)

| Check | Observable | Result |
|-------|-----------|--------|
| AC58a | K_ESCAPE KEYDOWN in START → `state` stays `START` | **PASS** |
| AC58b | K_ESCAPE KEYDOWN in GAME_OVER → `state` stays `GAME_OVER` | **PASS** |
| AC58c | No `K_ESCAPE → return False` pattern in `_handle_events()` | **PASS** |
| AC58d | `pygame.QUIT` event → `_handle_events()` returns `False` (app quit intact) | **PASS** |

The old `if event.key == pygame.K_ESCAPE: return False` line is confirmed absent. The new
K_ESCAPE handler only acts when `state is PLAY` (→ PAUSE) or `state is PAUSE` (→ PLAY);
all other states fall through the `# else: START or GAME_OVER — silent no-op` path.
`pygame.QUIT` handler at the top of the event loop is unchanged and returns `False`.
