### ★ THE DEFECT — AC120 / R132 (the smoke path never reaches DAMAGING)
**`SMOKE_TIMELINE` ordering bug.** The LASER is seeded at frame **1** (WINDUP = `BEAM_WINDUP_F` 30 f →
DAMAGING would arm ~frame 31), but the scripted **bomb at frame 20** runs `bombs._flush`, which (correctly)
clears `world.beams`. So the seeded beam is wiped **mid-WINDUP at frame 20** and **never reaches the
DAMAGING phase** in the headless run. Frame-by-frame trace of the real `App(smoke=True).run()`:
`f1–19 = ['WINDUP']`, `f20 = []` (bomb flush, charges 2→1, flash armed), `f21–120 = []`. **`any DAMAGING ==
False`.** R132/AC120 explicitly require the smoke path to exercise a **full WINDUP→DAMAGING cycle including
widening, sweeping, surviving a contact, and a beam-attributed kill** — the bomb destroys all of that.
- Why the unit suite missed it: `test_smoke_seed_runs_full_cycle` calls `seed_smoke_laser` then
  `update_beams` 30× **in isolation** (no App, no bomb), so it confirms the seed *can* reach DAMAGING but
  not that the *actual smoke run* does. The new e2e
  `test_v20_laser_attribution.py::test_smoke_run_exercises_full_laser_cycle` drives the real loop and fails.
- **Cheapest fix (Programmer, 1-line config):** reorder `SMOKE_TIMELINE` so the laser cycle completes
  before the bomb, or re-seed the laser **after** frame 20 (e.g. laser @ frame 21 → DAMAGING ~f51, timeout
  ~f81; or move the bomb later). A laser whose 90 f WINDUP+DAMAGING window doesn't overlap the f20 flush is
  all that's needed. Keep the bomb-flush-clears-beams behaviour (it is correct — see AC114-adjacent below).
  Re-run the new e2e witness to green.
