## V10.1 Feature in one line

Holding **Q** for the same **30 frames (0.5 s)** that already quits from PAUSE now also quits from the
**START** and **GAME_OVER** screens, drawing the **same v8 progress arc** while held; a **single shared
hold counter** is **reset to zero on every state transition** so a partial hold can never leak across
screens and cause an early quit; Q in **PLAY** still does nothing.

## V10.2 ★ v8 REUSE — t_quit, arc visual, gesture semantics (R76/R77/R80) — CONFIRMED VERBATIM

The §38 lever **"Confirm v8 reuse"** is resolved: the gesture reuses the shipped v8 mechanic
**exactly**, with **NO per-state variation** of threshold, visual, or cancel logic. There is **one
deliberate, R80-mandated difference** — the *idle visibility* of the arc widget on the new screens — and
it is recorded explicitly in §V10.6; it is **not** a change to the gesture mechanic itself.

| Reused element | v8 source | v10 value (identical on START / PAUSE / GAME_OVER) |
|---|---|---|
| **Hold threshold `t_quit`** | `v8-pause.md` §V8.3 | **`PAUSE_QUIT_FRAMES = 30` frames = 0.5 s @ 60 FPS** — same constant, **no new value, no per-state variation** |
| **Arc shape** | §V8.4 | Circular arc, **clockwise from 12 o'clock**, sweeping 0°→360° as fill 0.0→1.0 |
| **Arc radius / stroke** | §V8.4 | **r = 22 px** (`PAUSE_ARC_R`), **stroke 3 px** (`PAUSE_ARC_STROKE`) |
| **Arc fill / track colour** | §V8.4 | fill **`HP_AMBER (240,190,50)`**, track **`HP_BACK (40,44,58)`** — **no new palette entry** |
| **Fill mapping** | §V8.4 / hud.py | `fill = q_hold_frames / PAUSE_QUIT_FRAMES`; CW draw `start = π/2 − fill·2π`, `end = π/2` |
| **Cancel-on-release** | R72 | Releasing Q before 30 f resets the counter **to 0 instantly**; the arc disappears; no action taken |
| **Restart-from-zero / no accumulation** | R72 | Each fresh press starts the counter at 0; interrupted holds **never accumulate** |
| **Quit action at full** | R72 | At `q_hold_frames >= PAUSE_QUIT_FRAMES`: `pygame.quit()` then `sys.exit(0)` — same teardown as PAUSE |

**Deviation register (required by §38):** the only deviation from the v8 PAUSE behaviour is *when the
arc widget is on screen* — see §V10.6. The threshold, geometry, colours, fill mapping, and cancel/reset
semantics are byte-for-byte the v8 values; nothing per-state is re-tuned.

## V10.3 ★ ACTIVE-STATE SET (R81) — LOCKED

The gesture (and its arc) is evaluated in exactly these states:

| State | Q-hold gesture active? | Arc drawn? | Notes |
|---|---|---|---|
| **START** | **YES** (new in v10) | while Q held (§V10.6) | counter evaluated in the START main-loop branch |
| **PLAY** | **NO** (R81) | never | Q is unbound in PLAY; the counter is **not** evaluated; a stray Q during a run can never quit. Mid-run quit still routes via Esc→PAUSE→hold Q |
| **PAUSE** | **YES** (v8, unchanged) | always-track + fill (v8) | shipped v8 behaviour is left exactly as-is |
| **GAME_OVER** | **YES** (new in v10) | while Q held (§V10.6) | counter evaluated in the GAME_OVER main-loop branch |

**Active set after v10: START + PAUSE + GAME_OVER.** PLAY is excluded — this is the R81 spine and the
§40 "PLAY accidental quit" guard. In the code today the counter is incremented **only** in the PAUSE
branch of the main loop (`app.py` ~line 184); v10 adds the **same** increment/reset block to the START
and GAME_OVER branches and adds **none** to PLAY. The existing `App._q_held()` seam (`app.py` ~line 154)
— real `pygame.key.get_pressed()[K_q]` live, scripted `held` set headless — is reused unchanged for all
three active states, so the new paths stay testable under the v9 `--event-script` gate.

## V10.4 ★ THE RESET-ON-TRANSITION SPINE (R79) — LOCKED (the correctness centre of v10)

There is **one** Q-hold counter — `App.q_hold_frames` (an `int`, UI state on `App`, **not** on
`World`) — consulted in START, PAUSE, and GAME_OVER. It **MUST be reset to 0 on every game-state
transition** so a partial hold accrued in one state can never carry into another and trigger an
immediate/early quit, and the arc can never appear pre-filled on a freshly entered screen.

**Every transition, named, with its required reset and current code status** (`app.py`):

| # | Transition | Where (today) | Reset to 0 today? | v10 requirement |
|---|---|---|---|---|
| 1 | **START → PLAY** (start the run) | `app.py` ~L94–95 | **NO** | **ADD `self.q_hold_frames = 0`** |
| 2 | **PLAY → PAUSE** (Esc) | `app.py` ~L87–89 | **YES** (L89) | keep |
| 3 | **PAUSE → PLAY** (Esc resume) | `app.py` ~L90–91 | **NO** | **ADD `self.q_hold_frames = 0`** |
| 4 | **PLAY → GAME_OVER** (player dies) | `app.py` ~L74–76 | **NO** | **ADD `self.q_hold_frames = 0`** ← the #1 risk: dying with Q incidentally held must **not** instantly quit |
| 5 | **GAME_OVER → PLAY** (R restart) | `app.py` ~L96–98 | **NO** | **ADD `self.q_hold_frames = 0`** |
| 6 | **PAUSE → PLAY** (R restart) | `app.py` ~L99–102 | **YES** (L101) | keep |

Transitions 2 and 6 already reset (shipped v8); transitions **1, 3, 4, 5 must gain the reset in v10**.
Transition 4 (PLAY→GAME_OVER) is the load-bearing one called out in requirements §40 and AC66: the
counter must zero **on entry to GAME_OVER**, not merely on key release, so that holding Q at the moment
of death cannot bleed a near-full counter into the GAME_OVER screen and quit on frame one. The initial
state is START with `q_hold_frames = 0` set in `App.__init__` (`app.py` ~L32) — already correct.

**Design rule for the Programmer (one sentence):** *every assignment to `self.state` is immediately
paired with `self.q_hold_frames = 0`.* Treating "set state" and "zero the hold counter" as a single
atomic step is the simplest way to guarantee R79 across all six transitions and any future one.

## V10.5 ★ START "ANY KEY STARTS" — Q CARVE-OUT (integration decision) — LOCKED

**The one place the gesture collides with existing behaviour, and the Designer's resolution.** Today,
START transitions to PLAY on **any** KEYDOWN (`app.py` ~L94: `if self.state is GameState.START:
self.state = GameState.PLAY`). That includes **Q** — so a Q key-press would start the game on frame one,
and the held-counter (which advances per *held frame*, not on the KEYDOWN edge) could never reach 30.
Left unchanged, **R76 (hold Q to quit from START) would be unreachable.**

**LOCKED resolution:** on the START screen, **Q is removed from the "any key starts" trigger** and
reserved for the hold-to-quit gesture. Every **other** key still starts the run.

```python
# app.py _handle_events(), START branch (was: if self.state is GameState.START:)
if self.state is GameState.START and event.key != pygame.K_q:
    self.q_hold_frames = 0        # transition #1 reset (§V10.4)
    self.state = GameState.PLAY
```

- This is a Designer state-machine decision (the gesture-vs-state-machine integration), not a value to
  delegate — it is the START analogue of the v8 §V8.2 Esc state-table lock.
- **No analogous carve-out is needed for GAME_OVER or PAUSE.** GAME_OVER only acts on a `K_r` KEYDOWN
  (`app.py` ~L96); Q does nothing on its edge there, so the held-counter drives the quit cleanly. PAUSE
  is v8, unchanged.
- **Writer note (not a blocker):** the START prompt copy ("press any key…", Writer-owned) now has a Q
  exception. Pressing exactly Q to *start* is vanishingly rare, so the prompt can stay; the new START
  quit-hint (R78, Writer) is what teaches that Q is the hold-to-quit key. Flagged so nothing is left
  hunting — Writer decides the wording.

## V10.6 ★ ARC ON START + GAME_OVER — idle visibility + placement constraint (R80)

### V10.6.1 Idle visibility — the one recorded deviation from v8 PAUSE (LOCKED)

In v8 PAUSE the empty arc **track** is drawn *always* (it is a permanent element of the pause menu,
`hud.py` ~L229). On the **START** and **GAME_OVER** screens there is no persistent menu panel, so a
permanently-visible amber ring with no obvious purpose would be confusing. Per **R80** ("the arc is
drawn only while Q is being held and disappears immediately on release/quit") and **AC68** ("drawing
them *with and without* the arc present"):

- **START / GAME_OVER:** render the **whole arc widget (track ring + fill arc) only while `q_hold_frames
  > 0`** (i.e. while Q is held). When Q is not held, **nothing** is drawn — the screen is visually
  unchanged from today. The fill mapping and geometry are the v8 values (§V10.2).
- **PAUSE:** **unchanged** — leave the v8 always-on track exactly as shipped. Do **not** touch
  `draw_pause`.

This is the deviation registered in §V10.2: it changes only *when* the widget appears on the two new
screens, never the threshold, geometry, colours, or cancel/reset semantics. It is mandated by R80, not a
free per-state restyle.

### V10.6.2 Placement constraint for the Artist (the Artist sets the pixels)

The Artist owns the **exact arc centre `(x, y)` on each of the two screens** (§38). The Designer fixes
only the constraints and the intent; the literal coordinates and any final font-clearance arithmetic are
the Artist's, validated by the v9 render-smoke gate.

**Hard constraints (objective pass/fail = the v9 render-smoke key-rect check, AC68):**

1. The arc's bounding rect (a 44×44 box, `r=22`) must **not overlap any text rect** on that screen.
2. The arc must be **clearly visible** as a quit-progress indicator (AC64) — on START against the dark
   `BG`, on GAME_OVER against the existing α=160 dim; `HP_AMBER` reads on both, **no new palette entry**.
3. Horizontal centring at `x = W//2 = 300` is the natural default (matches every other full-screen
   element); the Artist may offset if a layout needs it, as long as constraints 1–2 hold.

**Existing text rects to clear (current layouts, for the Artist's reference):**

- **START** (`hud.draw_start`, no dim): TITLE y=250 (FONT_BIG) · PITCH y=320 (FONT_SMALL) · CONTROLS_1
  y=470 · CONTROLS_2 y=500 · START_PROMPT y=560 (blinking). **Open vertical bands:** ≈ **y 345–460**
  (between the pitch and the controls block) and ≈ **y 590–800** (below the prompt). *(Note: the v10
  Writer adds a START quit-hint line, R78 — its final y is Writer/Artist-coordinated; the arc must clear
  it too once placed.)*
- **GAME_OVER** (`hud.draw_gameover`, α=160 dim): GAMEOVER_TITLE y=300 (FONT_BIG) · SCORE y=380
  (FONT_MID) · BEST y=420 (FONT_MID) · GAMEOVER_KEYS y=480 (FONT_SMALL, the line the Writer extends with
  the quit hint, R78). **Open vertical band:** ≈ **y 505–800** (below the key list).

**Design intent (non-binding, Artist may improve):** place the arc **near / just below the screen's
quit-hint line** so the player reads "the amber ring = how full my hold-Q-to-quit is." A centred arc in
the lower open band of each screen satisfies this and the render-smoke gate. Final pixels = Artist.

## V10.10 Programmer implementation guide

All changes are in the existing module set; no new files. The gesture/arc/threshold logic already exists
(v8) — v10 generalises *where* it runs and guarantees the reset.

### `app.py` — `_handle_events()`
1. **START carve-out + reset (§V10.5, transition #1):**
   `if self.state is GameState.START and event.key != pygame.K_q:` → `self.q_hold_frames = 0;
   self.state = GameState.PLAY`.
2. **PAUSE→PLAY (Esc) reset (transition #3):** add `self.q_hold_frames = 0` in the `elif self.state is
   GameState.PAUSE: self.state = GameState.PLAY` Esc branch.
3. **GAME_OVER→PLAY (R) reset (transition #5):** add `self.q_hold_frames = 0` to the `GAME_OVER and
   K_r` restart branch.
   *(PLAY→PAUSE and PAUSE→PLAY-via-R already reset — leave them.)*

### `app.py` — `_step_play()`
4. **PLAY→GAME_OVER reset (transition #4, the #1 risk):** where `w.player.hp <= 0` sets
   `self.state = GameState.GAME_OVER`, add `self.q_hold_frames = 0` so a Q held at death cannot leak
   into GAME_OVER.

### `app.py` — main loop
5. **Generalise the hold-counter block to the active set (§V10.3, R81).** The v8 PAUSE block
   (`if pressed: q_hold_frames += 1; if >= PAUSE_QUIT_FRAMES: pygame.quit(); sys.exit(0); else:
   q_hold_frames = 0`) must run when **`state in (START, PAUSE, GAME_OVER)`** — **not** in PLAY. Reuse
   `self._q_held()` for the "pressed" test (live keyboard or scripted `held` set). Suggested shape:
   ```python
   if self.state in (GameState.START, GameState.PAUSE, GameState.GAME_OVER):
       if self._q_held():
           self.q_hold_frames += 1
           if self.q_hold_frames >= C.PAUSE_QUIT_FRAMES:
               pygame.quit(); sys.exit(0)
       else:
           self.q_hold_frames = 0
   elif self.state is GameState.PLAY:
       self._step_play(inp)
   ```
   (PLAY stays mutually exclusive — the counter is never advanced in PLAY, R81/AC67.)

### `app.py` — `_draw()`
6. **START:** after `hud.draw_start(...)`, call the new arc draw **only while `q_hold_frames > 0`**
   (§V10.6.1), at the Artist's START arc centre.
7. **GAME_OVER:** after `hud.draw_gameover(...)`, call the new arc draw **only while `q_hold_frames >
   0`**, at the Artist's GAME_OVER arc centre.
   (PAUSE's `draw_pause` is untouched.)

### `view/hud.py`
8. Factor the v8 arc draw (`draw_pause` lines that draw track + CW fill, `hud.py` ~L223–244) into a
   small reusable helper, e.g. `draw_quit_arc(screen, center, q_hold_frames)`, and call it from
   `draw_pause` (unchanged behaviour: track always) and from two new tiny screen helpers
   `draw_start_quit_arc` / `draw_gameover_quit_arc` (or a guarded call) that draw **only when
   `q_hold_frames > 0`**. Reuse `PAUSE_ARC_R`, `PAUSE_ARC_STROKE`, `HP_AMBER`, `HP_BACK`,
   `PAUSE_QUIT_FRAMES` verbatim — **no new arc constants.**

### `config.py`
9. Add **nothing** for the Designer. Writer adds the two quit-hint strings (R78); the two arc-centre
   coordinates can live as Artist-named consts (e.g. `START_ARC_CENTER`, `GAMEOVER_ARC_CENTER`) per the
   Artist's spec, or be passed inline — Artist/Programmer choice.
