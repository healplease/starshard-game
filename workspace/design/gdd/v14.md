## V14.1 Decision in one line

The lifetime stats live on a **new first-class `STATS` GameState** reached **only from START** by
pressing **Tab**, and left by pressing **Tab** (toggle) or **Esc** (back) to return to START. No run is
running while STATS is shown; no other key acts.

## V14.2 ★ Placement — a new STATS GameState (not a GAME_OVER panel, not a PLAY toggle) — LOCKED

| Option considered | Verdict |
|---|---|
| **New `STATS` state off START** | **CHOSEN.** START is the natural "home/menu" — keyboard-only, no run in progress, single full screen free for the 5-row ledger. Mirrors how v8 §V8.2 made PAUSE a first-class state. |
| Panel on GAME_OVER | **Rejected.** GAME_OVER is already full — TITLE y=300, SCORE y=380, BEST y=420, KEYS y=480, plus the Q-arc and R-arc near y=505–545 (`v10`/`v12`). Five more rows won't fit, and lifetime totals are a *between-runs* concern, not an end-of-run one. (GAME_OVER's **BEST** line already surfaces the persisted highscore — see V14.6.) |
| Toggle overlay during PLAY | **Rejected.** Collides with Esc=pause; reading stats mid-run makes no sense and would need a freeze the menu already gives for free. |

`STATS` is a peer of START/PLAY/PAUSE/GAME_OVER in the `GameState` enum. It renders a full screen (the
starfield may scroll behind it as on START — Artist's call) and runs **no world update** (no spawns, no
collisions, no scoring) — it is a static view of the R94 in-memory store.

## V14.3 ★ Navigation & keybindings — LOCKED (honors every existing binding)

| From | Key | To | Notes |
|---|---|---|---|
| **START** | **Tab** (`K_TAB`) | **STATS** | enter the ledger |
| **STATS** | **Tab** (`K_TAB`) | **START** | toggle back out |
| **STATS** | **Esc** (`K_ESCAPE`) | **START** | "back" — Esc never quits (v8 R73 stands); it backs out of the sub-screen, like PAUSE→PLAY |

- **No other key acts in STATS.** Any key besides Tab/Esc is ignored (it does **not** start a run and
  does **not** quit) — so a stray keypress can't dump the player out unexpectedly.
- **Entry is from START only.** Not reachable from PLAY, PAUSE, or GAME_OVER. One canonical path keeps
  the state machine flat.
- **Why these keys:** Tab is unused anywhere in the game and reads as "switch view"; Esc already means
  "back out of a sub-state" (PAUSE→PLAY) and is a no-op on START today, so reusing it as STATS→back adds
  no collision. Neither is a movement key, so neither overloads WASD/arrows.

## V14.4 ★ START "any-key-starts" — Tab carve-out (the one integration decision) — LOCKED

Today START transitions to PLAY on **any** KEYDOWN except **Q** (the v10 §V10.5 carve-out). Tab would
therefore *start the run* instead of opening STATS. **Resolution (mirrors v10 §V10.5 exactly):** on
START, **Tab is removed from the "any key starts" trigger** and routes to STATS; **Q** stays carved out
for hold-to-quit; **every other key still starts the run.**

```python
# app.py _handle_events(), START branch — extend the v10 Q carve-out with Tab
if self.state is GameState.START:
    if event.key == pygame.K_q:
        pass                                  # v10: reserved for hold-Q-to-quit
    elif event.key == pygame.K_TAB:
        self.q_hold_frames = 0; self.r_hold_frames = 0   # transition reset (V14.5)
        self.state = GameState.STATS
    else:
        self.q_hold_frames = 0; self.r_hold_frames = 0   # v10/v12 START→PLAY reset
        self.state = GameState.PLAY
```

This is a state-machine decision (gesture-vs-any-key integration), not a value to delegate — the START
analogue of the v10 Q carve-out and the v8 §V8.2 Esc table.

## V14.5 ★ Reset-on-transition spine — extend to the two new transitions (hygiene)

The v10 §V10.4 / v12 §V12.4 spine rule — *every `self.state = …` is paired with `q_hold_frames = 0` AND
`r_hold_frames = 0`* — applies to the two new transitions too:

| # | Transition | Reset |
|---|---|---|
| 7 | **START → STATS** (Tab) | zero both counters |
| 8 | **STATS → START** (Tab or Esc) | zero both counters |

Q-hold and R-hold are **not** active in STATS (V14.7), so this is hygiene, not a live-bug guard — but
keeping the invariant universal means no future "Q held across the menu" surprise and matches the
existing code shape.

## V14.6 Highscore readout — already on GAME_OVER; STATS shows the full ledger

GAME_OVER's existing **BEST** line (`hud.draw_gameover`, y=420) is the highscore; under v14 it now reads
the **persisted** store (R92) instead of a session-only value — a Programmer wiring detail, no layout
change. The STATS screen is the single place showing **all five** lifetime values **+ highscore**
together (R97/AC84). The two are consistent (same `highscore` field); STATS is the full ledger, BEST is
the at-a-glance highscore where the player already looks.

## V14.7 Active sets unchanged — STATS is NOT added to the Q/R hold gestures

- **Q-hold-to-quit** active set stays **START + PAUSE + GAME_OVER** (v10 §V10.3) — STATS is **not** added.
  To quit, the player presses Esc/Tab back to START (one key away) and holds Q there. Keeps v14 minimal:
  no new arc, no new active-state branch.
- **R-hold-to-restart** active set stays **PAUSE + GAME_OVER** (v12 §V12.8) — nothing to restart from a
  menu reached out of START.
- Consequently STATS draws **no** quit/restart arc and needs no arc placement from the Artist.
