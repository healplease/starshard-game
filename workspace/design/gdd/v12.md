## V12.1 Feature in one line

On **PAUSE** and **GAME_OVER**, holding **R** for the same **30 frames (0.5 s)** that already quits
on Q now **restarts the run** (`world.reset_run()` → PLAY), drawing the **same v8 progress arc** while
held; the **restart counter is fully independent of the quit counter** (holding R never fills the
quit arc and vice-versa); **both** counters are **reset to zero on every state transition**; and the
gesture is active **only** in PAUSE + GAME_OVER (never START, never PLAY).

## V12.2 ★ THRESHOLD REUSE (R83/R84) — CONFIRMED: reuse the v8 value, coupled at one source

The §44 lever **"Confirm threshold reuse"** is resolved: the restart hold uses the **same threshold
as quit** — **no distinct restart value**.

| Lever (§44) | Decision | Programmer constant |
|---|---|---|
| **Hold duration to restart `t_restart`** | **30 frames = 0.5 s @ 60 FPS — reused from v8 verbatim** | **`RESTART_HOLD_FRAMES = PAUSE_QUIT_FRAMES`** (= 30) |

**Decision + rationale (the deviation register §44 asks for — there is *no* deviation):**
- `t_restart` **equals** `t_quit` so the two gestures **feel identical** (the brief's explicit goal:
  "so quit and restart feel symmetric"). A player who has learned the ~0.5 s Q-hold instantly knows
  the R-hold.
- **Coupling, not just copying.** Lock `RESTART_HOLD_FRAMES = PAUSE_QUIT_FRAMES` as a **self-documenting
  alias bound to the single source of truth**, rather than a second literal `30`. This (a) keeps the
  restart code readable — `r_hold_frames >= C.RESTART_HOLD_FRAMES` does not pretend to be a *quit*
  threshold — and (b) guarantees the two gestures stay symmetric if quit is ever retuned. If a future
  increment ever *wants* them to diverge, split the alias into its own literal **then**; today they
  are deliberately one value.
- **Fill mapping (reused from §V8.4 / §V10.2):** `fill = r_hold_frames / RESTART_HOLD_FRAMES`
  (0.0 → 1.0), drawn clockwise from 12 o'clock exactly as the Q arc.
- **Programmer latitude:** inlining `C.PAUSE_QUIT_FRAMES` for the R comparison is acceptable, but the
  alias is recommended for readability. **No new threshold number is introduced.**

## V12.3 ★ TWO INDEPENDENT HOLD COUNTERS (R86) — LOCKED (the #1 correctness item)

There are now **two** integer hold counters, both **UI state on `App`** (not on `World`):

| Counter | Gesture | Active states | Threshold | Action at full |
|---|---|---|---|---|
| `q_hold_frames` (existing, v8/v10) | Q-hold-to-**quit** | START, PAUSE, GAME_OVER | `PAUSE_QUIT_FRAMES` | end loop → `pygame.quit()` |
| **`r_hold_frames`** (NEW, v12) | R-hold-to-**restart** | **PAUSE, GAME_OVER only** | `RESTART_HOLD_FRAMES` | `world.reset_run()` → PLAY |

**Independence contract (R86 / AC72):**
- Holding **R** advances **only** `r_hold_frames` and fills **only** the restart arc. It **MUST NOT**
  touch `q_hold_frames` or the quit arc.
- Holding **Q** advances **only** `q_hold_frames` and fills **only** the quit arc. It **MUST NOT**
  touch `r_hold_frames` or the restart arc.
- Holding **both** keys simultaneously advances **both** counters **independently**, each on its own
  per-frame `+1` / reset-on-release; whichever reaches its threshold first fires its action.
- Releasing **one** key resets **only that** counter (cancels only that gesture); the other is
  **unaffected**.

**Implementation rule (one sentence):** each counter is driven by its **own** held-key test
(`_q_held()` for Q, a new **`_r_held()`** seam for R) and its **own** `+= 1` / `= 0` branch — they
**never read each other's key state and never share a variable**. This is the structural guarantee
against the §46 "counter cross-talk" risk.

### V12.3.1 Tie-break when BOTH complete on the same frame (degenerate case) — LOCKED

If a player holds **both** Q and R continuously for the full 30 frames, both counters reach threshold
on the **same** frame. **Quit takes precedence** (the more terminal action): the quit block is
evaluated **first** in the main loop and ends the run, so the app exits — the (also-complete) restart
never matters because the process is quitting anyway. This is a non-goal edge (normal play holds one
gesture at a time); it is locked only so the ordering is deterministic, not ambiguous.

## V12.4 ★ BOTH COUNTERS RESET ON EVERY STATE TRANSITION (R88) — LOCKED (the correctness spine)

This **extends the v10 §V10.4 single-counter spine to the second counter.** Every game-state
transition **MUST reset BOTH `q_hold_frames` AND `r_hold_frames` to 0**, so a partial hold accrued in
one state can never carry into another and trigger an immediate/early action, and **neither arc can
appear pre-filled** on a freshly entered screen.

**Every transition, named, with its current `q` reset status and the v12 `r` requirement** (`app.py`):

| # | Transition | Where (today) | `q_hold_frames=0` today? | v12 requirement |
|---|---|---|---|---|
| 1 | **START → PLAY** (any key ≠ Q) | `app.py` ~L99–101 | **YES** (v10) | **ADD `r_hold_frames = 0`** (defensive — R isn't active in START, but pair it) |
| 2 | **PLAY → PAUSE** (Esc) | `app.py` ~L88–90 | **YES** (v10) | **ADD `r_hold_frames = 0`** |
| 3 | **PAUSE → PLAY** (Esc resume) | `app.py` ~L91–93 | **YES** (v10) | **ADD `r_hold_frames = 0`** |
| 4 | **PLAY → GAME_OVER** (player dies) | `app.py` ~L74–77 | **YES** (v10) | **ADD `r_hold_frames = 0`** ← entering GAME_OVER with R incidentally held must **not** instantly restart |
| 5 | **PAUSE → PLAY** (R-restart) | **CHANGES** — moves from the `K_r` KEYDOWN branch (~L106–109) to the main-loop R-hold block (§V12.5) | n/a (rewritten) | the restart trigger sets `state=PLAY` and **resets BOTH** counters atomically |
| 6 | **GAME_OVER → PLAY** (R-restart) | **CHANGES** — moves from the `K_r` KEYDOWN branch (~L102–105) to the main-loop R-hold block (§V12.5) | n/a (rewritten) | the restart trigger sets `state=PLAY` and **resets BOTH** counters atomically |

**Transitions 1–4** already zero `q_hold_frames` (the v10 spine); v12 just pairs an
`r_hold_frames = 0` beside each. **Transitions 5 & 6 are *rewritten* this increment** (the restart
moves off the KEYDOWN edge — §V12.5); the new restart trigger zeroes **both** counters as part of the
restart so a partial R-hold cannot survive into PLAY and "double-fire" on the next screen (the §46
"both-counters-reset leak" risk, AC73).

**Design rule for the Programmer (one sentence):** *every assignment to `self.state` is immediately
paired with `self.q_hold_frames = 0` **and** `self.r_hold_frames = 0`.* Treating "set state" and
"zero both hold counters" as one atomic step guarantees R88 across all six transitions and any future
one. `r_hold_frames` is initialised to 0 in `App.__init__` alongside `q_hold_frames`.

## V12.5 ★ RESTART BECOMES A HELD GESTURE — KEYDOWN branches removed (R83/R84) — LOCKED

This is the **activation-model change** and the structural heart of v12. Today R-restart fires on a
**`K_r` KEYDOWN edge** inside `_handle_events` (two branches: GAME_OVER `app.py` ~L102–105, PAUSE
~L106–109). After v12, R-restart is driven by the **held counter in the main loop**, exactly as v8
moved quit off the Esc KEYDOWN edge onto the Q-hold counter.

**LOCKED:**
1. **Remove both `K_r` KEYDOWN restart branches** from `_handle_events` (the PAUSE one and the
   GAME_OVER one). A single R key-press no longer restarts. *(This is the v12 analogue of the v8 R73
   "remove the `K_ESCAPE → return False` line" ruling.)*
2. **Add an R-hold block in the main loop**, parallel to the existing Q-hold block, active when
   `state in (PAUSE, GAME_OVER)`: each held frame `r_hold_frames += 1`; on reaching
   `RESTART_HOLD_FRAMES`, run `world.reset_run()`, zero **both** counters, and set `state = PLAY`; on
   release, `r_hold_frames = 0` (cancel — §V12.6).
3. **`reset_run()` semantics are unchanged** — restart is behaviourally identical to the current
   PAUSE restart (v8 R74) and GAME_OVER restart (v1 R13): same `world.reset_run()` → directly to
   **PLAY** (not through START, not back through PAUSE). v12 changes **only how restart is armed**
   (press → hold), not what it does.
4. A **sub-threshold tap of R** (held < 30 frames) does **nothing** — the counter resets on release
   before it reaches threshold.

**No confirmation prompt beyond the arc** (R-hold arc *is* the guard, replacing the v8 R74 one-step
instant restart — §45 non-goal).

## V12.6 ★ CANCEL-ON-RELEASE / NO ACCUMULATION FOR R (R87) — LOCKED

Identical to the v8 R72 / §V10.2 cancellable rule, applied to `r_hold_frames`:
- Releasing **R** at any point before `RESTART_HOLD_FRAMES` elapses **cancels** the gesture:
  `r_hold_frames` resets **to 0 instantly**, the restart arc disappears, and **no restart occurs**
  (the screen stays in its current state).
- Pressing R again starts the counter **from 0** — there is **no accumulation** across interrupted
  holds.
- This is the main-loop `else: r_hold_frames = 0` branch (the `not _r_held()` case), mirroring the
  existing Q `else` branch.

### V12.7.2 Idle-visibility — the R arc MATCHES the Q arc on the same screen (LOCKED)

The existing Q arc already has **different** idle-visibility per screen: on **PAUSE** the empty track
ring is drawn **always** (it's a permanent element of the pause panel, v8 §V8.6.1); on **GAME_OVER**
the whole arc widget is drawn **only while Q is held** (v10 §V10.6.1, no persistent menu). The R arc
**adopts the Q arc's rule on each screen**, so the two gestures look consistent side-by-side:

| Screen | Q arc (existing) | **R arc (v12) — LOCKED** |
|---|---|---|
| **PAUSE** | track always on; fill while held | **track always on; fill while `r_hold_frames > 0`** (matches Q) |
| **GAME_OVER** | whole widget only while held | **whole widget only while `r_hold_frames > 0`** (matches Q) |

This satisfies R88/AC73 "neither arc appears **pre-filled** on a freshly entered screen": the *fill*
is always 0 on entry (both counters reset); an empty *track* on PAUSE is a static menu element, not a
fill. (Artist note: PAUSE now hosts **two** always-on empty tracks — place them so neither overlaps
the other or any text, §V12.7.3. If two static rings read as clutter, the sanctioned fallback is to
draw the PAUSE R track only-while-held too — but the primary ruling is *match the Q arc per screen*.)

### V12.7.3 Placement constraint for the Artist (the Artist owns the pixels)

The Artist sets the **exact R-arc centre `(x, y)` on PAUSE and on GAME_OVER** (§44). The Designer
fixes only the constraints; the literal coordinates and font-clearance arithmetic are the Artist's,
validated by the v9 render-smoke gate (AC77).

**Hard constraints (objective pass/fail = the v9 render-smoke rect check, now covering two arcs):**
1. The R-arc's 44×44 bounding rect (r=22) must **not overlap the Q arc** — **PAUSE Q-arc centre

2. It must **not overlap any text rect** on that screen (PAUSE: title + the three hint lines;

3. It must be **clearly visible** as its own progress indicator (`HP_AMBER` reads on the PAUSE α=110

**Design intent (non-binding, Artist may improve):** place the R arc **beside / just below the "hold
R  Restart" hint line** so the player reads "this amber ring = how full my hold-R-to-restart is,"
mirroring how the Q arc sits by the quit hint. Two arcs sharing a screen must read as two distinct
gauges — separate them clearly (the Q arc is centred at x=300 on both screens, so the R arc will
typically want a horizontal **or** vertical offset of its own). Final pixels = Artist, gated by
render-smoke.

## V12.8 ★ START / PLAY EXCLUSION (R90) — LOCKED

The R-hold gesture is evaluated in **exactly** PAUSE + GAME_OVER:

| State | R-hold active? | R arc drawn? | Notes |
|---|---|---|---|
| **START** | **NO** | never | START has no Restart (it already starts on any key); the R-hold block does **not** run here, so a stray R neither arms an arc nor restarts. *(The v10 Q START carve-out is unaffected — Q still reserved for hold-to-quit on START; R simply isn't in the START active set.)* |
| **PLAY** | **NO** | never | R is unbound in PLAY; the counter is **not** evaluated; a stray R during a run can never restart. Mid-run restart still routes **Esc→PAUSE→hold R**. |
| **PAUSE** | **YES** (new) | per §V12.7.2 | counter evaluated in the main-loop R block |
| **GAME_OVER** | **YES** (new) | per §V12.7.2 | counter evaluated in the main-loop R block |

**Active set for the R gesture after v12: PAUSE + GAME_OVER.** This is narrower than the Q gesture's
START+PAUSE+GAME_OVER set (Q can quit from START; R has nothing to restart there). The R-hold block's
state guard is therefore `state in (PAUSE, GAME_OVER)` — **not** the Q block's
`(START, PAUSE, GAME_OVER)`. Keeping the two blocks' state guards distinct is what enforces R90
(§46 "START/PLAY accidental restart" risk).

## V12.11 Programmer implementation guide

All changes are in the existing module set; no new files. The threshold + arc + cancel + reset logic
all **already exist** (v8/v10) — v12 adds a **parallel counter**, a **second arc**, moves restart
**off the KEYDOWN edge**, and scopes it to **PAUSE + GAME_OVER**.

### `app.py` — `__init__`
- Add `self.r_hold_frames = 0` beside `self.q_hold_frames` (UI state on `App`, not `World`).

### `app.py` — `_handle_events()`
- **Remove** the two `K_r` KEYDOWN restart branches (the `GAME_OVER and K_r` branch ~L102–105 and the
  `PAUSE and K_r` branch ~L106–109). A single R press no longer restarts (§V12.5). *(Leave the Esc,
  START-carve-out, and X-bomb branches untouched.)*

### `app.py` — `_step_play()` (transition #4)
- Where death sets `state = GameState.GAME_OVER` (~L74–77, already zeroes `q_hold_frames`), **add
  `self.r_hold_frames = 0`** so entering GAME_OVER with R incidentally held can't instant-restart
  (§V12.4 #4).

### `app.py` — `_handle_events()` Esc + START transitions (#1, #2, #3)
- Beside each existing `self.q_hold_frames = 0` (START→PLAY ~L100, PLAY→PAUSE ~L90, PAUSE→PLAY-Esc
  ~L93), **add `self.r_hold_frames = 0`** (§V12.4 #1–3).

### `app.py` — `_r_held()` (new seam, mirrors `_q_held()`)
```python
def _r_held(self):
    if self.event_script is not None:
        return pygame.K_r in self.event_script.get("held", {}).get(self.frame, ())
    return bool(pygame.key.get_pressed()[pygame.K_r])
```

### `app.py` — main loop (the R-hold block, parallel to the Q block — transitions #5, #6)
After the existing Q-hold block (which stays exactly as-is), add:
```python
# v12: R-hold-to-restart — active in PAUSE + GAME_OVER only (independent counter)
if running and self.state in (GameState.PAUSE, GameState.GAME_OVER):
    if self._r_held():
        self.r_hold_frames += 1
        if self.r_hold_frames >= C.RESTART_HOLD_FRAMES:
            self.world.reset_run()
            self.q_hold_frames = 0          # transitions #5/#6: zero BOTH atomically
            self.r_hold_frames = 0
            self.state = GameState.PLAY
    else:
        self.r_hold_frames = 0
```
Notes: (a) the `running and …` guard enforces the §V12.3.1 **quit-precedence** tie-break (if the Q
block already ended the loop this frame, skip restart); (b) keep the existing
`if self.state is GameState.PLAY: self._step_play(inp)` — after an R-restart sets `state=PLAY` it may
step one frame this same frame, which is exactly the prior behaviour of the old KEYDOWN restart, so no
behavioural change; (c) the R block's state guard is `(PAUSE, GAME_OVER)` — **not** the Q block's
`(START, PAUSE, GAME_OVER)` — enforcing R90 (§V12.8).

### `app.py` — `_draw()`
- **PAUSE:** the R arc is part of the pause overlay (track always on, fill while held) — fold it into
  `hud.draw_pause` (pass `r_hold_frames`), or call a guarded draw after it, at the Artist's PAUSE R-arc
  centre.
- **GAME_OVER:** after `hud.draw_gameover`, draw the R arc **only while `r_hold_frames > 0`**
  (mirroring the existing `hud.draw_gameover_quit_arc` call), at the Artist's GAME_OVER R-arc centre.

### `view/hud.py`
- Generalise the v8/v10 arc helper to **`draw_hold_arc(screen, center, hold_frames, threshold)`**
  (the BA-suggested `draw_hold_arc(center, frac)` shape) and call it for **both** the Q and R arcs,
  passing the matching counter + threshold. Reuse `PAUSE_ARC_R`, `PAUSE_ARC_STROKE`, `HP_AMBER`,
  `HP_BACK` verbatim — **no new arc constants.** Keep `draw_pause`'s PAUSE-always-on-track behaviour
  for both arcs; keep the GAME_OVER arcs only-while-held (§V12.7.2).

### `config.py`
- Add **`RESTART_HOLD_FRAMES = PAUSE_QUIT_FRAMES`** (§V12.10). The two R-arc centre coords are
  Artist-named consts (e.g. `PAUSE_RESTART_ARC_CENTER`, `GAMEOVER_RESTART_ARC_CENTER`); the two
  restart-hint strings are the Writer's (R85).
