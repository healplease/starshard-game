## V16.1 vision (one line)
Every boss-spawn event still fires on the same cadence and runs the same arrival/freeze/reward loop — but
the game now **rolls a die** and a **different boss may show up**: today either the **Mothership** or a new,
**ship-free but deadlier** boss, with the pool built so a **third boss is one entry away**.

## V16.2 functional requirements (MoSCoW, testable)

Continues from R98 (R99+). "Shapes + text only / keyboard only / single screen" still applies (C1, C2).
The new boss **reuses the v7 encounter machinery** (R101) — this increment adds **one new Boss type + a
selection step**, not a new encounter system.

### A — The random, extensible boss pool (MUST)

- **R99 — Extensible boss pool (registry).** The bosses eligible to appear live in a **single boss pool /
  registry** (a data-driven list of boss types). Today it holds **exactly two**: the **Mothership** (v7)
  and the **new v16 boss**. **Adding a future boss MUST be a single registry entry** — registering one new
  boss type, with **no change** to the selection logic (R100) or the v7 encounter loop (R101). The pool is
  the **one** place the roster is defined; nothing downstream hard-codes "there are two bosses".

- **R100 — Uniform-random selection at each boss-spawn event.** Each boss-spawn event (the R56 breakpoint)
  selects **one** boss **uniformly at random** from the pool — every member has **equal probability** (1/N
  for N bosses; today ½ / ½). Selection is **independent per event**: consecutive bosses **may repeat**
  (no "no-repeat" / shuffle-bag rule this increment — pure i.i.d. uniform). The choice is made **once per
  encounter**, at spawn, and fixes which boss runs that fight. The RNG **MUST be seedable/overridable** so
  smoke + pytest can force a specific boss deterministically (R105).

- **R101 — Cadence + v7 encounter framing UNCHANGED for every boss.** The boss-appearance **cadence is
  unchanged** (first ~75 s, then +90 s — the v7/level_spec value) and the **full v7 encounter contract
  applies to whichever boss is selected**: arrival field-clear (R57), spawn-freeze while alive (R58),
  slow entrance → centre → oscillation (R59), large HP + always player-hittable (R60), ram + bullet damage
  to the player (R61), big flat reward + normal-loop resume on defeat (R62), bomb-vs-boss rule (R63), and
  the boss HUD bar + label (R64). The new boss **plugs into this same loop** — v16 changes only the
  **selection** of the boss, not the loop around it. (Per-boss values — HP, reward, moveset, visuals,
  name — naturally differ; the *framing* does not.)

### B — The second boss as new content, under two hard constraints (MUST)

- **R102 — A second boss type (new content).** A **second boss** joins the pool (R99) as a full boss: it
  obeys the v7 encounter contract (R101) and is a **distinct boss** from the Mothership in identity, shape,
  name, and attacks. Its **identity/theme, moveset/attack patterns, HP, defeat behaviour, reward, shapes,
  palette, name, and copy** are **delegated** (Designer/Artist/Writer/Level-designer) — subject to the two
  hard constraints below, which **override** any conflicting v7 default.

- **R103 — HARD CONSTRAINT 1 (human, non-negotiable): the new boss spawns NO ships/enemies.** The new boss
  **MUST NOT spawn ships, enemies, or minions of any kind** at any point in its fight. Spawning
  reinforcements is the **Mothership's** gimmick (v7 R67: 5 REGULAR / 2 HEAVY / 7 SCOUT) and is **exclusive
  to it**. The new boss's entire threat comes from **its own attacks/projectiles**, never from adds. The v1
  asteroid + v5 enemy spawners stay frozen as usual (R58); with the new boss the **only** hostiles on screen
  are **its own projectiles** (zero ships). *(This is a per-boss override of v7 R67, which is Mothership-only
  by definition — R67 is not a contract on all bosses.)*

- **R104 — HARD CONSTRAINT 2 (human, non-negotiable): the new boss's attacks are DEADLIER than the
  Mothership's.** The new boss's attacks **MUST pose a strictly higher threat** than the Mothership's —
  via **more damage per hit, and/or denser/more-frequent fire, and/or harder-to-dodge patterns** (the
  exact lever is the Designer + Level-designer's choice). "Deadlier" must be made **concrete with numbers**
  recorded in the GDD + level_spec (e.g. higher bullet damage than the Mothership's `EB_DMG`-based shots,
  or a denser/faster bullet pattern) so QA can verify the new boss is **measurably more dangerous** than the
  Mothership — not merely cosmetically different. It must still be **survivable/dodgeable** (the fight is
  winnable; "deadlier" ≠ "unfair/undodgeable") — balancing within that is the Level-designer's call.

### C — Smoke + tests (MUST)

- **R105 — Smoke gate green + both bosses + the pool exercised headlessly; suite grows.** After v16,
  `main.py --smoke-test` still **exits 0 after exactly 120 frames** headless (R14/R33/R43/R54/R65 preserved)
  with **no AC1–AC85 regression**. Because selection is random, smoke/tests **MUST be able to force a
  specific boss** (seed/override per R100) so the run can **deterministically** exercise (a) the **new
  boss's** arrival-clear + entrance + ≥1 attack step, and (b) that it **spawns no ships** (R103). The
  **pytest suite stays green and grows** with checks for: the **extensible pool** (R99), **uniform-random
  selection** (R100), the **new boss's no-ship constraint** (R103), and its **deadlier-than-Mothership**
  attack values (R104) — while **Mothership parity** (AC39–AC52) and AC1–AC85 still hold.

> v16 is acceptance-checked against R99–R105 (all MUST). No COULD items this increment.

## V16.4 non-goals (still out of scope)
- ❌ Changing the boss **cadence** or any other part of the v7 encounter framing — only *which* boss appears
  is randomized (R101).
- ❌ **Weighted / no-repeat / shuffle-bag** selection — v16 is **pure uniform i.i.d.**; repeats are allowed
  (R100). (Weighting is a possible later increment.)
- ❌ **Multiple simultaneous bosses** — still **one boss at a time** (v7 §27 stands); the pool picks **one**
  per event.
- ❌ The new boss **spawning ships/enemies/minions** of any kind — **forbidden** (R103). Adds remain the
  Mothership's exclusive mechanic.
- ❌ Boss **phases / enrage / health-gated forms** — each boss keeps **one** HP pool + **one** repeating
  moveset (v7 §27 stands).
- ❌ External assets — the new boss + its projectiles are **shapes + text** only (C2); no audio.
- ❌ Mouse / gamepad — keyboard-only (C1); controls unchanged.

## V16.5 risks / watch-items
- **"Deadlier" left vague.** If R104 isn't pinned to concrete numbers, QA can't verify it and "deadlier"
  degrades to "just different". → Designer/Level-designer **must record the lever + values** (§V16.3) so
  AC92 has something to measure against the Mothership.
- **Accidental ship spawn.** The new boss's moveset must be **projectile-only**; reusing any v7 R67
  minion-spawn path would violate R103. → AC91 checks **zero** ships across a full new-boss fight.
- **Selection not testable.** A non-seedable RNG makes the new boss / pool unverifiable headlessly (random
  runs may never pick it). → R100/R105 require a **force-a-boss** seed hook; AC87/AC93 lean on it.
- **Hard-coded "two bosses".** If selection or the encounter loop assumes exactly two bosses, R99
  extensibility breaks. → AC86 checks adding a third boss is a **one-entry** change with no logic edits.
- **Mothership regression.** Routing the Mothership through the new selection layer must not change its v7
  behaviour. → AC93 re-checks Mothership parity (AC39–AC52) + full AC1–AC85.

Continues AC numbering (AC86+). **[Smoke]** = headless via `--smoke-test`; **[Play]** = by observation;

- **AC87 [Code/Smoke]** Each boss-spawn event selects **one** boss **uniformly at random** from the pool

  **seedable** so a specific boss can be forced deterministically in tests (R100).
- **AC88 [Play/Smoke]** The boss-appearance **cadence is unchanged** — first boss at ~75 s, then +90 s,
  exactly as v7 — regardless of which boss is selected (R101).
- **AC89 [Play/Smoke]** Whichever boss is selected, the **full v7 encounter framing runs**: arrival
  field-clear (no charge/no score), spawn-freeze while alive then resume on defeat, slow entrance → centre

- **AC91 [Play/Smoke/Code]** **HARD CONSTRAINT 1:** across a complete new-boss fight, the new boss spawns

- **AC93 [Smoke]** `main.py --smoke-test` **exits 0 after exactly 120 frames** headless, the **pytest suite

  with **no AC1–AC85 regression** (R105).
