## The governing principle (read this first)

> **A kill/destroy counts only at the exact code site where the entity is destroyed *by player fire* —
> the site that already calls `scoring.award` for that entity.** This single rule resolves every edge
> case below, because the live code already scores exactly those events and silently despawns everything
> else. **Counting is separate from writing:** counters increment at their event site during play (R93);
> the file is written only at a flush (R95). Counting ≠ writing, so multiple flushes never double-count.

## R92 — Persistent save store (single JSON file, versioned)

A single JSON file persists five lifetime values across process runs. It lives in a **per-user,
per-game subfolder** (e.g. Windows `%APPDATA%\Starshard\`; exact path + cross-platform choice =
Programmer). The file carries a **`version`** integer for future migration. One JSON object at the file
root, UTF-8. **MUST**.

### Schema (v14 = version 1)

```json
{
  "version": 1,
  "highscore": 0,
  "runs": 0,
  "enemies_killed": 0,
  "asteroids_destroyed": 0,
  "bosses_killed": 0
}
```

| Field | Type | Meaning | Increment rule |
|---|---|---|---|
| `version` | int | schema version; **1** this increment | constant; bump only on a schema change |
| `highscore` | int ≥ 0 | best single-run score ever achieved | see R93 §highscore |
| `runs` | int ≥ 0 | number of runs begun | +1 each time a run begins (R93 §runs) |
| `enemies_killed` | int ≥ 0 | enemy ships shot down | +1 per enemy bullet-kill (R93 §enemies) |
| `asteroids_destroyed` | int ≥ 0 | asteroids/debris shot apart | +1 per asteroid bullet-kill (R93 §asteroids) |
| `bosses_killed` | int ≥ 0 | Motherships defeated | +1 per boss defeat (R93 §bosses) |

Field names are **snake_case and frozen** — code reads/writes exactly these keys. New future values =
new keys + a `version` bump + a migration step; never rename or repurpose an existing key.

## R93 — Exact increment semantics for the five values

All counts are **entity counts, unaffected by the Score×2 buff** (the multiplier touches `highscore`
only). Counters live on the process-lifetime store (R94), not the per-run world.

- **`asteroids_destroyed`** — +1 at the asteroid-destroy site in `systems/combat.py` step 1 (the
  `a.hits <= 0` branch that calls `scoring.award(a.score)`). One count **per asteroid removed**, not per
  hit — a large asteroid that takes 2 player bullets counts **1**. **Does NOT count:** asteroids cleared
  by a bomb flush or a boss-arrival clear (silent `.clear()`, no award), asteroids consumed by ramming
  the player, or asteroids that drift off-screen.

- **`enemies_killed`** — +1 at the enemy-destroy site in `combat.py` step 2 (the `e.hp <= 0` branch that
  calls `scoring.award(e.score)`). Counts all v5 ship kinds **REGULAR / HEAVY / SCOUT**, **including
  boss-spawned minions** (they are ordinary scored Enemies — if you shot it down, it counts). **Does NOT
  count:** enemies cleared by a bomb flush or boss-arrival clear, or enemies consumed by ramming the
  player (a collision-consume, no award).
  - **GREEN → RED split children:** **never count.** A GREEN pellet and its RED children are enemy
    *bullets* (projectiles), not enemy ships — the player cannot "kill" them, and they never flow through
    the enemy-destroy site. Likewise the boss's YELLOW→12-RED ring is bullets, not kills. Only the HEAVY
    ship that *fired* a pellet is an enemy; destroying that ship counts 1.

- **`bosses_killed`** — +1 in `systems/encounter.py::on_defeat` (boss `hp <= 0`, the only place a boss
  dies; it awards `BOSS_KILL_SCORE`). A boss is its own entity (not a v5 Enemy) → it counts **only** here
  and **never** toward `enemies_killed`. A boss **cannot** be bomb-killed (immune by construction), so
  there is no bomb path to count.

- **`runs`** — +1 **the moment a run begins**: the transition into PLAY with a freshly-initialized world
  (the world/score reset). This fires **once on the initial START→PLAY** and **once on every hold-R
  restart→PLAY** (each restart is a new run). It does **NOT** fire on: app launch, Esc-resume from PAUSE
  (same run continues — no world reset), GAME_OVER, or quitting from START (no run was ever begun).
  - **A run quit or restarted mid-game still counts**, because it was already counted *at its start*. The
    increment sits in the in-memory store (R94) and is persisted at the next flush (R95). Example:
    start (runs→1), die → GAME_OVER (flush writes 1), hold-R restart (runs→2), pause, hold-Q quit (flush
    writes 2) = **2 runs**, correct.

- **`highscore`** — the best single-run score the store has ever seen. At **every flush** (R95), set
  `highscore = max(stored highscore, current world score)`. On GAME_OVER that captures the run's final
  score; on a quit-from-PAUSE it captures the abandoned run's current (real) score; on quit-from-START it
  is `max(hs, 0)` = unchanged. Score only grows within a run, so the flush captures its high-water mark.
  The Score×2 buff is already baked into `world.score`, so a doubled score is reflected. `highscore`
  **never decreases**.

## R94 — In-memory accumulation, process-lifetime, restart-independent

The five values load once at startup (R92/R96) into a **single process-lifetime store**, NOT the
per-run world. They accumulate across the whole session. **A hold-R restart resets the world (score,
enemies, etc.) but MUST NOT reset these lifetime counters** — they carry across every run in the
session. **MUST.**

## R95 — Flush-trigger contract (write only on GAME_OVER + hold-Q quit)

The file is written **only** at these two moments, never otherwise (writes are expensive — kickoff):

1. **GAME_OVER** — on the frame the run transitions PLAY→GAME_OVER (player hp ≤ 0).
2. **hold-Q quit confirmed** — the instant the quit gesture completes, **before the process exits**, from
   any quit-capable state (START / PAUSE / GAME_OVER).

Each write: refresh `highscore` per R93, then **atomically** serialize the full store (all 5 values +
`version`) and replace the file (write-temp-then-rename, so a crash mid-write never corrupts the file).
**No** per-event writes, **no** timer writes, **no** write on pause/resume or on restart. Multiple
flushes are idempotent (e.g. GAME_OVER then quit-from-GAME_OVER re-writes the same store — harmless,
because counting is decoupled from writing, R93). A hard crash / force-kill that bypasses both paths may
lose the current session's unflushed increments — **accepted** (kickoff). **MUST.**

## R96 — Missing / corrupt / partial file → fall back to zeros, never crash

Loading the store **MUST NOT crash** under any file condition. Fallback rules:

- **File missing / unreadable** → start all five counts at **0**, `version` = current (1). (First flush
  then creates the file.)
- **Unparseable JSON, or root is not a JSON object** → treat as corrupt → all five = **0**, version = 1.
- **Per-field defensive recovery** (a partially-valid file recovers what it can): for each of the five
  count fields, if it is missing, not an integer, or negative → default **that field** to 0; otherwise
  keep its value. `version`: if missing or not an int, or an unrecognized value (≠ a known schema
  version) → treat the file as unknown and fall back to all-zeros rather than trust an unknown shape.
- Under no circumstance does a bad save file raise to the player or prevent the game from starting. The
  next flush overwrites the file cleanly. **MUST.**

The five lifetime values (runs, enemies killed, asteroids destroyed, bosses killed, + highscore) are
**displayed on-screen** in a lifetime-stats view — single-screen, keyboard-only, placeholder text/shapes
only (no new assets). **This R only requires that the values are shown.** *Where* the screen sits in the

## R98 — Smoke gate stays green; save I/O is headless-safe

The `--smoke-test` gate stays green (120 frames, exit 0). The smoke/headless run **MUST NOT mutate the
real user save file** — the save path must be **overridable** (env var or arg) so smoke/regression runs
target a temp/throwaway location, leaving the player's real lifetime stats untouched. **MUST.**

## Acceptance criteria (AC78–AC85)

- **AC78 — fresh install.** No save file present → game loads with all counts 0 and does not crash; the
  first flush creates the file containing exactly the six keys (`version` + five values).
- **AC79 — round-trip persistence.** After a flush, the five values + highscore re-load equal in a new
  process (write → relaunch → read back identical).
- **AC80 — corrupt/partial file.** A garbage or partially-broken file → loads as zeros (or per-field
  recovery per R96), no crash; the next flush overwrites it cleanly.
- **AC81 — flush only on the two triggers.** The file is written on GAME_OVER and on hold-Q quit, and
  **not** on pause/resume, restart, or any per-event tick (observable: the file is unchanged across a
  pause/resume and across a hold-R restart that does not end a run).
- **AC82 — count accuracy.** Shooting N asteroids apart → `asteroids_destroyed` += N (large = 1 each);
  shooting enemies (incl. boss minions) → `enemies_killed` rises per kill; bomb/arrival clears and
  ram-consumes do **not** increment either; a boss defeat → `bosses_killed` += 1 (and **not**
  `enemies_killed`); `runs` increments once per run begun (initial start + each restart), and a
  quit/restarted-mid-game run still counts.
- **AC83 — highscore.** `highscore` = max over flushes, reflects a Score×2-boosted score, and never
  decreases across runs in a session or across processes.
- **AC84 — stats screen renders.** The lifetime-stats view displays all five values + highscore without
  error (render-smoke gate; exact placement/labels per Designer/Artist/Writer).
- **AC85 — headless-safe I/O.** A `--smoke-test` run does not modify the real user save file (uses the
  overridable temp path).
