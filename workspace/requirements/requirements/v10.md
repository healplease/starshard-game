## 37. v10 functional requirements (MoSCoW, testable)

Continues numbering from v8 (R76+). "Shapes + text only / keyboard only / single screen" still
applies (C1, C2). The gesture's mechanics are **frozen by reuse of the shipped v8 implementation** —
no new threshold, arc visual, or cancel logic is invented; v10 only changes **which states** can run it
and **where the hint/arc appear**.

### A — Enable the gesture in two more states (MUST)

- **R76 — Q-hold-to-quit on the START screen.** While in the **START** state, holding **Q**
  continuously for the v8 threshold **t_quit** (`PAUSE_QUIT_FRAMES = 30 f`) quits the application —
  behaviourally identical to holding Q in PAUSE (R72): the application tears down pygame and exits the
  process. The same v8 progress arc grows from empty (0 %) to full (100 %) over t_quit and reaching
  full triggers the quit.

- **R77 — Q-hold-to-quit on the GAME_OVER screen.** While in the **GAME_OVER** state, holding **Q**
  continuously for **t_quit** quits the application, identical to R76 / R72. The same arc shows
  progress.

- **R78 — Quit hint visible on both screens.** The Q-hold-to-quit gesture **MUST** be discoverable on
  both screens:
  - The **START screen** must teach the gesture (a "hold Q  Quit" hint), alongside its existing
    start/controls copy.
  - The **GAME_OVER screen** key list (`GAMEOVER_KEYS`, currently `"R  Restart"` after v8 — **no quit
    hint**) must add the quit hint.
  The **exact strings** are the **Writer's** (§38); this requirement only fixes that the hint must be
  present and accurate on both screens. No stale "Esc Quit" copy may be (re)introduced (v8 R73 still
  holds — Esc never quits).

### B — Shared hold-counter correctness across states (MUST)

- **R79 — One shared hold counter, reset on every state transition.** There is a **single** Q-hold
  progress counter (the v8 one), now consulted in START, PAUSE, and GAME_OVER. It **MUST reset to
  zero on every game-state transition** so that a partial hold accrued in one state can **never**
  carry into another and cause an immediate/early quit. In particular:
  - Entering GAME_OVER (player dies) with Q incidentally held must **not** instantly quit — the
    counter starts at zero on the new screen and only accrues from a fresh continuous hold.
  - Restarting from GAME_OVER (R13) or from PAUSE (R74) into PLAY must reset the counter to zero.
  - Any START↔PLAY↔PAUSE↔GAME_OVER transition resets it to zero.
  This generalises the v8 cancellable/no-accumulation rule (R72) across the multi-state surface.

### C — Arc rendered on the new screens (MUST)

- **R80 — Arc rendered on START + GAME_OVER (reuse the v8 arc).** The v8 progress-arc draw is invoked
  on the START and GAME_OVER screens while Q is held, using the **same visual** (shape/size/colour) as
  v8. Its **screen position on each of these two screens** is delegated to the **Artist** (§38) and
  **must not collide** with that screen's existing text (title, hints, score, key list). The arc is
  drawn only while Q is being held and disappears immediately on release/quit (consistent with v8).

### D — PLAY exclusion preserved (MUST)

- **R81 — Q-hold-to-quit is NOT active in PLAY.** The gesture remains **inactive during active play**:
  holding Q in PLAY does **not** arm the arc and does **not** quit (Q is unbound in PLAY, unchanged
  from v8 §33). Quitting mid-run still goes through PAUSE (Esc → PAUSE, then hold Q). This keeps an
  accidental Q press during a run from ever ending the game. (Active states for the gesture after v10:
  **START, PAUSE, GAME_OVER**.)

### E — Smoke / render gates (MUST)

- **R82 — Smoke + render-smoke preserved; no scripted-input change.** `main.py --smoke-test` still
  **exits 0 after exactly 120 frames** headless (SDL dummy drivers); the Q key is **never held** on the
  scripted smoke path, so the gesture is a **no-op** for the smoke run (no new seeding or scripted
  events). The **new arc-draw paths on START + GAME_OVER MUST pass the v9 render-smoke gate** — drawing
  the START and GAME_OVER screens (with and without the arc present) raises no exception, and the
  arc's bounding rect does **not overlap** the screens' key/text rects (the v9 "key rects don't
  overlap" check). **No AC1–AC60 regression.** The Programmer confirms and QA verifies (AC68).

## 40. v10 risks / watch-items

- **Cross-state hold-counter leak (the #1 risk).** If the hold counter is **not** reset on a state
  transition, dying or restarting while Q is incidentally held could **instantly quit** on the next
  screen, or the arc could appear pre-filled. R79 + AC66 guard this explicitly — it's the spine of the
  increment. The Programmer must reset on **every** transition, not just on key-release.
- **Arc ↔ text collision.** START and GAME_OVER already have title/score/key-list text; the reused arc
  must be placed so it doesn't overlap them (Artist, §38). AC68's render-smoke key-rect check is the
  objective gate; AC64 confirms visibility.
- **Hint accuracy / stale copy.** GAME_OVER's `GAMEOVER_KEYS` currently has **no** quit hint and START
  must teach the gesture; if either is left unupdated, the feature is undiscoverable. No "Esc Quit"
  string may reappear (v8 R73). AC65 checks both.
- **PLAY accidental quit.** If the gesture is wired too broadly (e.g. "any state") it could become
  active in PLAY and let a stray Q end a run. R81 + AC67 keep PLAY excluded.
- **Smoke false-comfort.** As in v8, the smoke gate passes trivially because Q is never held headlessly
  — that does **not** exercise the new code. The real new-code gate is the **render-smoke** on START +
  GAME_OVER (AC68) and live playtest (AC61–AC64). Don't read a green smoke as coverage of the gesture.

## 41. v10 acceptance criteria (what QA will check)

Continues AC numbering (AC61+). **[Smoke]** = headless via `--smoke-test`; **[Play]** = by
observation; **[Code]** = by code inspection. Headline — *holding Q on the START and GAME_OVER screens
quits the game with the same cancellable progress arc that already works in PAUSE, and both screens
show a quit hint* — is covered by AC61–AC68.

- **AC61 [Play]** Holding **Q on the START screen** for the full **t_quit** (`PAUSE_QUIT_FRAMES`)
  fills the progress arc to completion and then the **application exits** (R76).

- **AC62 [Play]** Holding **Q on the GAME_OVER screen** for the full **t_quit** fills the arc to
  completion and then the **application exits** (R77).

- **AC63 [Play]** On **either** screen, **releasing Q before t_quit** resets the arc immediately to
  empty and takes **no further action** (the screen stays as it was); pressing Q again starts from
  zero — **no accumulation** across interrupted holds (R76/R77 reuse of R72).

- **AC64 [Play]** The **progress arc is visible** while Q is held on the START screen and on the
  GAME_OVER screen, using the same visual as the v8 PAUSE arc (R80).

- **AC65 [Play/Code]** The **START screen shows a "hold Q  Quit" hint** and the **GAME_OVER key list
  shows a quit hint** added to `"R  Restart"`. Neither screen displays a stale "Esc Quit" string
  (R78, v8 R73).

- **AC66 [Code/Play]** The Q-hold counter is **reset to zero on every state transition** (START→PLAY,
  PLAY→PAUSE, PAUSE→PLAY, PLAY→GAME_OVER, GAME_OVER→PLAY restart, PAUSE→PLAY restart). A partial hold
  carried into a new state does **not** cause an immediate or early quit and the arc never appears
  pre-filled on a freshly entered screen (R79).

- **AC67 [Code/Play]** Q-hold-to-quit is **not active in PLAY** — holding Q during active play does
  not arm the arc and does not quit; mid-run quit still requires Esc→PAUSE then hold Q (R81).

- **AC68 [Smoke/Code]** `main.py --smoke-test` **exits 0 after exactly 120 frames** headless (SDL
  dummy drivers) with no scripted-input change, **and** the v9 **render-smoke passes on the START +
  GAME_OVER screens** (drawing them with and without the arc raises no exception, and the arc rect
  does not overlap the screens' text/key rects). **No AC1–AC60 regression** (R82).
