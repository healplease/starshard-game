# v8 increment — Pause / Unpause + Q-hold to Quit

Owner: business-analyst · Date: 2026-06-05 · Status: complete
Source: `workspace/shared/brief.md` (v8 increment, via backlog), `workspace/shared/backlog.md` v8

> This section **appends** to v1 + v2 + v5 + v6 + v7; it does not replace them. All prior
> requirements (R1–R68) and acceptance criteria (AC1–AC52) still hold. v8 adds a **PAUSE game
> state**, a **Esc-toggle** to enter/exit it, a **Q-hold-to-quit gesture** with a visible progress
> arc, a **pause-screen overlay** with control hints, and removes Esc as a direct-quit key.
> It continues numbering from R68 / AC52 (the next IDs are **R69** / **AC53**).
> Concrete numbers (hold threshold, arc style/placement, overlay layout, restart-from-pause key)
> are **delegated** to the Designer/Artist/Writer — see §32. The behavioral contract is fixed here.

## 30. v8 vision (one line)

Pressing Esc in PLAY pauses the game completely — everything freezes mid-action and a hint overlay
appears; a second Esc resumes; to quit, the player holds Q until a progress arc completes (~0.5 s),
which may be cancelled by releasing the key before it finishes.

## 31. v8 functional requirements (MoSCoW, testable)

Continues numbering from v7 (R69+). "Shapes + text only / keyboard only / single screen" still
applies (C1, C2). All existing machinery is unchanged by this increment unless explicitly noted.

### A — PAUSE state and game freeze (MUST)

- **R69 — PAUSE state: Esc toggle in PLAY.** In the PLAY state, a single **Esc KEYDOWN** event
  transitions the game to a new **PAUSE** state. A second **Esc KEYDOWN** while in PAUSE
  transitions back to PLAY. These are the **only** state transitions Esc triggers — in every other
  state (START, GAME_OVER) Esc is **silently ignored** (R73). The PAUSE state is a **new, first-
  class game state** alongside START, PLAY, and GAME_OVER (not a flag on PLAY).

- **R70 — Complete game freeze while PAUSE is active.** The moment the game enters PAUSE, **all
  game-state advancement stops**. Specifically (BA rulings — NOT delegated):
  1. **All entity positions and velocities are frozen** — player, asteroids, enemies (incl. the
     Mothership boss and all boss minions), and all in-flight projectiles (player bullets, all
     enemy bullets, split pellets, fan bullets, split children) hold their exact positions.
  2. **All spawners are frozen** — the v1 asteroid/debris spawner, the v5 enemy spawner, the v2
     bonus-pickup drip spawner, and the v7 boss-minion moveset steps all produce nothing.
  3. **The boss breakpoint run clock does NOT advance during PAUSE.** Pausing cannot be used to
     delay or skip an approaching boss breakpoint — the clock is frozen and resumes from exactly
     the same value when the player unpauses.
  4. **All buff duration timers freeze** — no buff expires or ticks down while paused.
  5. **All boss encounter timers freeze** — boss entrance position, oscillation phase, and moveset
     cadence timers all hold.
  6. **The player cannot act** — Z-fire, X-bomb, and movement inputs have no effect while PAUSE is
     active (the PLAY update loop does not run).
  7. **No score is awarded** while paused (no survival tick, no kill/pickup events).
  8. **Damage resolution stops** — no player-bullet × entity collisions and no entity-× player
     collisions are processed while paused.

  The behavioral contract: **unpausing and re-pausing the game at any instant yields exactly the
  same state** — no entity has drifted, no timer has changed. The Designer's GDD (§32) enumerates
  the specific subsystems to freeze; this list is the behavioral specification those sub-systems
  must satisfy.

  **Optionally delegated to Designer (§32):** whether the starfield scroll and cosmetic particles
  continue running during PAUSE (purely cosmetic — they cannot affect game state). All other items
  above are non-negotiable BA rulings.

### B — Pause-screen overlay (MUST)

- **R71 — Pause overlay with control hints.** While PAUSE is active, a **visible overlay** is
  rendered over the game world (the world is visible beneath it). The overlay **MUST** display at
  minimum:
  - A **PAUSED indicator** (label or heading — text exact wording delegated to Writer, §32).
  - A **resume hint** — the action and key to unpause (Esc → resume).
  - A **quit hint** — the Q-hold gesture and its approximate duration.
  - A **restart hint** — the key to restart the run from PAUSE (R74).
  The overlay is **removed instantly** when the player returns to PLAY (second Esc) or when the
  run is restarted (R74). The **exact layout, background, font sizes, colors, and placement** of
  the overlay are the **Artist's** spec (§32); the **exact text strings** for each hint are the
  **Writer's** (§32). The overlay must not be confused with the GAME_OVER screen (R11).

### C — Q-hold-to-quit gesture (MUST)

- **R72 — Q-hold-to-quit with cancellable progress arc.** While **PAUSE is active**, the player
  may quit the application by **holding Q continuously** for a threshold duration **t_quit**. The
  **exact value of t_quit** (target ≈ 0.5 s) is a **Designer value** (§32). Behaviour:
  - While Q is held: a **visible progress arc** (or progress indicator — style and exact position
    delegated to Artist, §32) grows from **empty (0 %)** to **full (100 %)** over t_quit. The arc
    must be visible on the pause overlay (R71) while growing.
  - When Q has been held continuously for the full t_quit: the application **quits permanently**
    (equivalent to closing the window — tears down pygame and exits the process).
  - If the player **releases Q at any point before t_quit elapses**: the gesture is **cancelled**,
    the arc **resets immediately to empty**, and no further action is taken. The game stays in
    PAUSE.
  - If Q is pressed, released early, then pressed again: the timer and arc **restart from zero**
    on each new press — there is no accumulation across interrupted holds.
  **BA ruling (NOT delegated):** Q-hold-to-quit is **only active in the PAUSE state**. It is not
  active in START, PLAY, or GAME_OVER (see §33, non-goal). The Designer may extend it to other
  states in a future increment — that is out of scope for v8.

### D — Esc no longer quits directly (MUST)

- **R73 — Esc removed from quit path in all states.** The K_ESCAPE KEYDOWN event **no longer
  triggers application quit in any game state**. Its only bindings after v8 are the PAUSE toggle
  (R69): **Esc in PLAY → PAUSE; Esc in PAUSE → PLAY**. In all other states (START, GAME_OVER),
  Esc is **silently ignored**. The existing `if event.key == pygame.K_ESCAPE: return False` in
  `app.py`'s `_handle_events` is **removed by the Programmer**. This is a **BA ruling — NOT
  delegated**; it is a hard requirement for v8. The pygame.QUIT event (window close button) is
  **unchanged** and continues to exit the application.

### E — Restart from PAUSE (MUST)

- **R74 — Restart from PAUSE.** From the PAUSE state, the player may **restart the current run**
  (reset to a fresh game). The behavior on activation is **identical to the GAME_OVER restart**
  (R13 / R31): `world.reset_run()` is called and the game transitions directly to **PLAY** (not
  back through PAUSE). The **key binding for this gesture** is a **Designer value** (§32); the
  pause overlay (R71) shows a hint for it (Writer, §32). BA ruling: restarting from PAUSE is a
  conscious destructive action and is acceptable without a secondary confirmation — the Q-hold arc
  is the only guarded gesture in this increment.

### F — Smoke gate (MUST)

- **R75 — Smoke gate preserved; no scripted-input change required.** After v8, `workspace/game/
  main.py --smoke-test` still **initializes pygame, runs exactly 120 frames with simulated input,
  and exits 0** under `SDL_VIDEODRIVER=dummy` / `SDL_AUDIODRIVER=dummy` — R14/R33/R43/R54/R65
  preserved, and **no AC1–AC52 regression**. The smoke path **never enters the PAUSE state**
  because: (a) `_handle_events()` is **not called** in smoke mode (the run loop uses
  `pygame.event.pump()` + `smoke_input()` directly), and (b) `smoke_input()` generates only
  `dx`, `dy`, and `fire` values — it **never synthesises a K_ESCAPE KEYDOWN event**. No new
  seeding, scripted Esc events, or changes to the 120-frame path are required for v8. The
  Programmer **confirms** this at implementation time and the QA tester **verifies** it (AC60).

## 33. v8 non-goals (out of scope)

- ❌ Q-hold-to-quit in START or GAME_OVER — Q-hold is PAUSE-only in this increment (R72 ruling).
  Extending it to other states is a future decision.
- ❌ A "Resume" or "Quit to Menu" button — keyboard-only (C1); no mouse/gamepad.
- ❌ Save / load / checkpoint — game state is held in memory only; PAUSE is a live freeze, not
  a checkpoint.
- ❌ A settings or options menu reachable from PAUSE.
- ❌ Any change to the START or GAME_OVER screen behavior beyond removing the Esc-quit binding
  (R73) and updating the hint text strings (Writer, §32).
- ❌ Multiple simultaneous pauses or nested pause states.
- ❌ An "are you sure?" confirmation on restart-from-PAUSE — R74 is a one-step action (see R74
  ruling).
- ❌ Any change to the Q key in PLAY or any other state — Q is only bound in PAUSE.
- ❌ Audio: all feedback is visual only (C2 — no external sound files).
- ❌ Any change to the smoke-test scripted path, frame budget, or seeding — confirmed no-op (R75).

## 34. v8 risks / watch-items

- **Freeze completeness.** If any timer or position-update slips through the freeze (R70), the
  game drifts between pause/resume cycles: boss breakpoint fires early, buff expires mid-pause,
  enemy drifts position. AC53 / AC54 check the behavioral contract; the Designer's freeze-list
  enumeration in the GDD is the implementation target.
- **Boss-mid-pause correctness.** The boss entrance, oscillation, and moveset timers must all
  freeze independently. If the oscillation uses `pygame.time.get_ticks()` or wall-clock instead of
  the world's frame counter, it won't freeze correctly — the Programmer must use the world frame
  clock exclusively (consistent with the rest of the engine).
- **Q-hold arc accumulation bug.** If the hold timer accumulates across interrupted key-presses
  (R72: "restarts from zero on each new press"), a player tapping Q three times could accidentally
  quit. AC56 and AC57 check the cancel / reset behavior explicitly.
- **Esc regression.** Removing `K_ESCAPE → return False` in `_handle_events` must be done
  carefully so it doesn't also remove the `pygame.QUIT` handler that shares the same loop (R73).
  AC58 checks that the window-close event still works.
- **Overlay/arc HUD collision.** The pause overlay and Q-hold arc must not visually collide with
  the boss health bar (AC47), buff pills (R29), or the player HP bar (R10) — Artist's layout
  responsibility (§32).
- **Controls copy left stale.** `config.py`'s `CONTROLS_2` currently reads `"QUIT  Esc"` and
  `GAMEOVER_KEYS` reads `"R  Restart      Esc  Quit"`. Both must be updated (Writer, §32);
  if not, players on the START / GAME_OVER screens will read incorrect control hints (AC59).
- **Smoke-gate false comfort.** The smoke gate not needing changes (R75) is a confirmed property
  of the architecture (event pump, not event handler, in smoke mode) — but if the Programmer adds
  any new `smoke_input`-path Esc injection, the PAUSE state would activate mid-smoke and halt the
  PLAY update loop, causing the 120-frame run to silently stop advancing game state. AC60 guards
  this.

## 35. v8 acceptance criteria (what QA will check)

Continues AC numbering (AC53+). **[Smoke]** = headless via `--smoke-test`; **[Play]** = by
observation; **[Code]** = by code inspection. The brief's headline — *Esc in PLAY pauses (freezes
all state); Esc again resumes; Q-hold for threshold quits with a visible cancellable arc; overlay
shows control hints; Esc never quits directly* — is covered by AC53–AC60.

- **AC53 [Play]** Pressing **Esc in PLAY** transitions the game to **PAUSE** — all entities and
  timers stop updating immediately; a visible pause overlay (R71) appears. A second **Esc in
  PAUSE** transitions back to **PLAY** with every entity in the exact position and state it held
  when paused (no drift, no spawn, no score change during the pause interval) (R69, R70).

- **AC54 [Play]** During a PAUSE that spans at least 2 seconds of real time, the **boss breakpoint
  run clock does not advance** — on resume, the next boss trigger fires at the same elapsed
  in-game time as it would have without the pause (R70 bullet 3).

- **AC55 [Play]** The **pause overlay is visible** while PAUSE is active; it displays at minimum
  a PAUSED indicator, an Esc-resume hint, a Q-hold-to-quit hint, and a restart hint; it is
  **removed instantly** on unpause or restart (R71).

- **AC56 [Play]** Holding **Q in PAUSE** for the full **t_quit** threshold causes the **progress
  arc to fill to completion** and then the **application exits** (R72).

- **AC57 [Play]** Holding **Q in PAUSE** then **releasing before t_quit** causes the **arc to
  reset immediately to empty** and the game to **stay in PAUSE** with no further action. Pressing
  Q again starts the arc from zero — there is **no accumulation** across interrupted holds (R72
  cancellable rule).

- **AC58 [Play/Code]** **Esc in START** does nothing (no state change, no quit). **Esc in
  GAME_OVER** does nothing (no state change, no quit). The `K_ESCAPE → return False` path in
  `app.py` **no longer exists**. The `pygame.QUIT` event (window close) **still exits** the
  application (R73).

- **AC59 [Play/Code]** The **START screen control hints** (`CONTROLS_2`) and **GAME_OVER screen
  control hints** (`GAMEOVER_KEYS`) no longer refer to "Esc Quit"; they reflect the updated
  control scheme (Esc = pause, Q hold = quit) per the Writer's revised strings (R73, §32).

- **AC60 [Smoke/Code]** `main.py --smoke-test` **exits 0 after exactly 120 frames** headless
  (SDL_VIDEODRIVER=dummy / SDL_AUDIODRIVER=dummy). Confirmed by code inspection: (a) `smoke_input()`
  returns only `dx`, `dy`, `fire` values — **no K_ESCAPE event is ever synthesised**; (b)
  `_handle_events()` is **never called** in smoke mode; therefore the PAUSE state is **never
  entered** headlessly. No new seeding or scripted-input changes are needed. **No AC1–AC52
  regression** (R75).
