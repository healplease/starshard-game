## V20a.2 The LASER enemy body — a stationary emitter / turret-eye (R120)

**Read story:** something that **parks and charges** — a squat, heavy, **forward-staring turret eye**,
not a darting fighter. Where REGULAR/HEAVY/SCOUT are pointed magenta polygons that *move and shoot
dots*, and the bosses are huge central bodies, the LASER is a **mid-size gunmetal housing with one big
glowing orange lens** aimed down-screen — it visibly *looks at you and charges*. The lens is the
emitter the beam springs from (it pre-reads the threat exactly like the Mothership's yellow core /
NOVA's white heart do).

| Property | Value | Why |
|---|---|---|
| **Painted silhouette** | **~34 w × 30 h** (rounded housing + a forward orange eye) | mid-size: bigger than a SCOUT dart, reads as a heavier installation; the **body ⊇ its collision circle** |
| **Collision (Level-designer locks `LASER_R`)** | circle **r ≤ 15** at center | the **housing half-extents (17 w, 15 h) ⊇ an r=15 circle** → no phantom hits. If LD locks r>15, widen the housing to match (keep body ≥ collision). Recommended `LASER_R = 14`. |
| **Housing** | `LASER_BODY` `#424656` filled octagon-ish block, 34×24, + 2 px `STAR_FAR` `#5A6482` outline | cold metal turret base; the grey-blue outline (not magenta) reinforces "installation, not fighter" |
| **Emitter eye** | `LASER_EYE` `#FF7A1A` filled circle **r=7** centered on the muzzle, + a `BEAM_CORE` `#FFF4E0` r=3 hot pupil | the single big orange lens = the danger tell + the beam origin (see §V20a.4); it **brightens during WINDUP** (the charge), see below |
| **Charging tell (WINDUP)** | during WINDUP only, draw a `LASER_EYE` ring r=9, width 2, around the eye | "powering up" — pairs with the windup line so the source of the telegraph is unmistakable |
| **Hit flash** | 1-frame `FLASH` fill on a non-lethal player-bullet hit (same as every other body) | consistency with v1/v5 |
| **Death burst** | v1 §5 — 6× `2×2` particles in the `LASER_BODY` fill colour, `[-3,3]` px/f, 20-f life | consistency |

### V20a.2.1 Body draw math (paste-ready) — center `(cx, cy)`, +y down

```python
LASER_HOUSE_W, LASER_HOUSE_H = 34, 24   # housing block (half: 17 × 12)
LASER_EYE_R   = 7                       # orange emitter lens radius
LASER_PUPIL_R = 3                       # white-hot pupil
LASER_EYE_DY  = 6                       # eye sits forward (toward +y, the muzzle side)
LASER_CHARGE_RING_R = 9                 # WINDUP "powering up" ring radius

# Housing — a squat octagon block (cut corners read "armored installation"), 34×24, chamfer 7.
laser_house = [(cx - 17, cy - 5), (cx - 10, cy - 12),   # left-top cut
               (cx + 10, cy - 12), (cx + 17, cy - 5),   # right-top cut
               (cx + 17, cy + 12), (cx - 17, cy + 12)]  # flat base (the muzzle face)
pygame.draw.polygon(screen, LASER_BODY, laser_house)
pygame.draw.polygon(screen, STAR_FAR,  laser_house, 2)          # cold grey-blue outline
# Emitter eye (the beam origin) on the muzzle face — orange lens + white-hot pupil.
ex, ey = cx, cy + LASER_EYE_DY
pygame.draw.circle(screen, LASER_EYE,  (int(ex), int(ey)), LASER_EYE_R)
pygame.draw.circle(screen, BEAM_CORE,  (int(ex), int(ey)), LASER_PUPIL_R)

    pygame.draw.circle(screen, LASER_EYE, (int(ex), int(ey)), LASER_CHARGE_RING_R, 2)
```

## V20a.3 The beam — WINDUP line vs DAMAGING beam (R122–R124, R127)

The beam is **one persistent line-segment instance per attack**, anchored at the emitter eye (§V20a.4),
running to **just past a screen edge** (R127). Two visual states match the two phases. **Critical
invariant (R124, GDD §V20.3.3): the beam's DRAW width == its COLLISION width — at every instant.**

### V20a.3.1 WINDUP — the thin, harmless telegraph line (R122/R123, harmless)

A **thin, faint, clearly-not-armed** orange line from the eye to just past the edge. It must read as a
*warning aim-line*, never as the lethal beam.

```python
WINDUP_W      = 2        # px — a THIN line (no collision; harmless per R123/AC111)
WINDUP_ALPHA  = 110      # ~43% — deliberately faint ("aiming, not firing")
# Draw on a per-pixel SRCALPHA surface (alpha baked into the colour — the v11 §V11.5 set_alpha gotcha),
# OR pygame.draw.line straight if no alpha is wanted; alpha is preferred so it reads "tentative".
# colour = LASER_EYE @ WINDUP_ALPHA  →  (255, 122, 26, 110)
pygame.draw.line(windup_surf, (*LASER_EYE, WINDUP_ALPHA), (ox, oy), (tx, ty), WINDUP_W)
```

- **Why thin + faint + alpha:** the whole skill is *read the tell and walk out* (GDD §V20.2). A faint
  2-px ghost line says "this is where the beam WILL be" without looking lethal — so a player who stays
  in it during windup (harmless, AC111) isn't unfairly punished by a scary-looking line. The orange
  hue ties it to the charging eye (same source).
- **No glow, no core** in windup — those belong to the armed beam. Pairs with the §V20a.2 charge ring
  on the eye so the *origin* visibly powers up while the *line* shows the *path*.

### V20a.3.2 DAMAGING — the wide, glowing, lethal beam that WIDENS (R123/R124, lethal)

When the beam arms, it becomes a **bright white-hot core sheathed in an orange glow**, and its width
**grows over the damaging phase** from a thin start to the final ~5–7 px the Level-designer locks. It
sweeps by **rotating about the eye** (GDD §V20.3.2(3)) — visually the near end stays pinned to the
lens, the far end swings; the drawing below redraws the segment at its current angle every frame, so
the rotation needs no special art handling.

**The widening is the headline visual** — it must *read* as "charging up to lethal". Drawn as **two
concentric strokes** along the same segment, both scaling with the live width `w`:

```python
# ── Level-designer locks these (defaults shown); Artist binds DRAW to them ──
BEAM_START_W  = 2        # px — the core width at the START of DAMAGING (just-armed, thin)
BEAM_FINAL_W  = 6        # px — FINAL core width at timeout (within R124's ~5–7 px; LD's exact call)
# live core width at progress t∈[0,1] through the damaging phase (curve = LD's; linear shown):
w = BEAM_START_W + (BEAM_FINAL_W - BEAM_START_W) * t      # the COLLISION width == this drawn core width

BEAM_GLOW_W      = 4        # px the orange glow extends BEYOND the core on EACH side (render-only halo)
BEAM_GLOW_ALPHA  = 90       # ~35% — soft outer halo

# 1) outer glow — orange, on an SRCALPHA surface (alpha baked in), wider than the core, render-only:
pygame.draw.line(glow_surf, (*LASER_EYE, BEAM_GLOW_ALPHA), (ox, oy), (tx, ty), int(w) + 2 * BEAM_GLOW_W)
#    (blit glow_surf to screen)
# 2) lethal CORE — white-hot, FULLY OPAQUE, width == w == the collision width (draw==collision, R124):
pygame.draw.line(screen, BEAM_CORE, (ox, oy), (tx, ty), max(1, int(round(w))))
```

- **The widen reads (R124/AC112):** the opaque white-hot **core** is the part that grows from 2 → ~6
  px over the damaging phase, so the beam visibly *thickens into lethality*; the orange glow tracks it
  (always `BEAM_GLOW_W` beyond each edge) so the whole bar swells together. A grazing pass across the
  early thin core can miss where the late-wide core connects — exactly the intended fairness window.
- **★ draw==collision invariant (R124 — the GDD CRITICAL INVARIANT):** **the lethal hit test uses the
  white-hot CORE width `w`, and the core is DRAWN at that same `w`** — one number drives both. The
  orange **glow is render-only and NOT part of collision** (it is a soft halo, like the v5 CYAN tail or
  the v16 bullet core — paint, not reach). So *what hits you is exactly the opaque white core you see*;
  the glow is honestly *outside* the kill zone. The Programmer computes `w` once per frame from the
  LD-locked start/final/curve and feeds **the same `w`** to both the segment/circle collision test
  (player `P_HITBOX_R` vs the segment at half-width `w/2`) and the core `draw.line` width. **Do not**
  introduce a separate draw width or pad collision to the glow.
- **Why white-hot core + orange glow:** a fully-opaque searing-white core is the universal "lethal
  energy beam" read and maximally separates the *armed* beam from the *faint orange windup line* (solid
  bright white vs faint thin orange) — you can never confuse "warning" with "firing". The orange glow
  binds it to the orange emitter eye (same weapon) and sells "heat" without enlarging the hit zone.

### V20a.3.3 Windup-vs-damaging at a glance (the two must never be confused)

| | WINDUP (harmless) | DAMAGING (lethal) |
|---|---|---|
| Core | none | **white-hot `#FFF4E0`, opaque** |
| Line colour | orange `#FF7A1A` @ **alpha 110** (faint) | white core + orange glow @ alpha 90 |
| Width | **2 px, fixed, thin** | **grows 2 → ~6 px** (LD's final, ~5–7) |
| Glow | none | orange halo, `BEAM_GLOW_W=4` px each side |
| Eye | charge ring (powering up) | full-bright lens (firing) |
| Collision | **none** (R123) | core width `w` (draw==collision) |

## V20a.4 Endless-to-edge geometry — origin anchor, which edge, overshoot (R127)

The beam is **enemy-anchored** (GDD §V20.3.2(3): it pivots about the enemy). The contract:

- **Origin anchor = the emitter EYE**, not the body center: `origin = (cx, cy + LASER_EYE_DY)`
  (`LASER_EYE_DY = 6`). The beam springs from the glowing lens, so the line visibly leaves the eye —
  the windup line, the armed beam, and the charge ring all share this one point. (The Programmer uses
  this same point as the geometric pivot for the rotation.)
- **Which edge / direction:** the line is aimed (at fire-time) **toward the fire-time player position**
  (GDD §V20.3.2(1)) — a generally **downward** ray (the LASER spawns from the enemy stream above the
  player). It extends from the eye **through that aim direction to the point where the ray crosses the
  nearest screen edge** (bottom or a side, whichever the ray reaches first), then **past it** by the
  overshoot below. As the beam rotates, recompute the far endpoint as that ray's edge-crossing each
  frame, so the line always runs cleanly off whatever edge it currently points at — it never stops
  short inside the field.
- **Overshoot past the edge:** `BEAM_OVERSHOOT = 24` px beyond the crossing point, so the far end is
  always drawn off-screen and the beam reads **endless** (no visible terminating dot inside or at the
  rim). 24 px > the widest glow (`~14 px`) so the rounded line-cap never peeks back on-screen.

```python
BEAM_OVERSHOOT = 24   # px past the screen-edge crossing → endless look (no terminating cap on-screen)
# origin = the emitter eye:
ox, oy = cx, cy + LASER_EYE_DY
# far end = extend the (current) aim ray to its first screen-edge crossing, then +BEAM_OVERSHOOT
# along the ray (Programmer's ray-vs-rect; collision clips the SEGMENT to the visible field as usual).
```

- **Collision still only spans the visible field:** the overshoot is purely paint — the segment used
  for the player hit test runs eye → edge-crossing (the off-screen tail can't touch the player anyway).
  Draw==collision still holds *on width*; the overshoot affects only the segment's far endpoint, which
  is off-screen.

## V20a.5 Render order — the beam slots into the enemy-bullet layer (extends §V19a.4)

The beam is the LASER's projectile, so it lives in the **enemy-bullet layer** (behind the player ship,
so a player standing in the beam still draws on top and reads as *in* it). The LASER body draws in the
**enemy layer** like every other enemy. **Within the enemy-bullet layer, draw the beam FIRST** (before
the RED/GREEN/CYAN/YELLOW/NOVA dots) so the small bullets sit on top of the wide beam and stay visible.

```
starfield → asteroids + ★BEAM (windup line / damaging glow+core)★ + enemy bullets
  → bonus diamonds → player bullets → enemies (incl. ★LASER turret★) → player (+ shield ring)
  → particles → hitbox indicator (PLAY+SHIFT) → HUD
```

This appends to the v19 §V19a.4 order; all other layers are unchanged. The beam's **alpha** surfaces
(windup line, damaging glow) use the v11 §V11.5 SRCALPHA rule (bake alpha into the draw colour; never
`set_alpha` a per-pixel surface); the **opaque white core** draws straight to screen.
