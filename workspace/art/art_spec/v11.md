## V11.1 What changes (one paragraph)

Today `_draw_player` **early-returns** — it skips drawing the ship entirely — on the "off" half of a
hard 6-frame half-cycle (`(blink_timer // 6) % 2 != 0`), so the ship strobes **255 → 0 → 255** alpha
(full → invisible). v11 replaces that hard on/off snap with a **smooth alpha pulse** between a **floor of
128/255 (~50%)** and a **ceiling of 255 (100%)**, eased on a **cosine** curve over a **30-frame full
cycle**. The ship **never fully disappears** (floor is 128, never 0), and the alpha varies smoothly every
frame instead of snapping between two states. The phase is driven by `p.blink_timer` so it still tracks
the remaining i-frames/Shield and the ship returns to solid the instant invulnerability ends.

## V11.2 The three locked alpha levers (paste as `config.py` constants)

```python
# ── Invulnerability pulse (v11 — supersedes the §V2.5 hard 6-f blink) ──────────
INVULN_ALPHA_FLOOR  = 128   # dimmest the ship reaches during invuln (~50% of 255). NEVER 0 — the ship
                            #   is always at least half-visible. Human spec "~50%" → locked at 128/255 (50.2%).
INVULN_ALPHA_CEIL   = 255   # brightest point of the pulse = fully opaque (the ship's normal solid look).
INVULN_PULSE_PERIOD = 30    # frames per ONE full pulse (bright→floor→bright). 0.5 s @60 FPS = 2 Hz —
                            #   a clear gentle "breathing" pulse, ~2.5× slower than the old 12-f strobe,
                            #   well below any flicker-fusion read. Replaces today's 6-f hard half-cycle.
```

| Lever | Value | Why |
|---|---|---|
| **Alpha FLOOR** | **128/255 (50.2%)** | Human said "~50%". 128 is the exact mid-point of [0,255]; the ship stays clearly visible at its dimmest, so the strobe is gentle on the eyes while still reading as "flashing / invulnerable." |
| **Alpha CEIL** | **255 (100%)** | The pulse peak is the ship's ordinary solid appearance — no over-bright, no new colour. |
| **PULSE PERIOD** | **30 f (0.5 s, full cycle)** | Old cadence was a 12-f full cycle (6-on/6-off ≈ 5 Hz). 30 f ≈ 2 Hz reads as a smooth pulse, not a flicker. Over the 1 s i-frame window (`IFRAMES=60`) the ship pulses **exactly twice**; over the 5 s Shield (300 f) it pulses 10× — both gentle. |
| **Curve** | **cosine (sine family)** | See §V11.3. Eases at *both* extremes → the smoothest "breathing" read; a triangle wave would still have a hard corner (a velocity reversal) at the floor and ceiling. |

## V11.3 The interpolation curve — cosine (the recipe)

**Curve choice: COSINE** (a sine-family ease), not a triangle. A triangle ramps linearly and reverses
direction sharply at the floor and ceiling — visibly "pointy". Cosine eases in *and* out at both extremes,
giving a soft breathing pulse — the gentlest possible read, which is exactly the human's intent ("pulses
rather than blinks").

Drive the phase off `p.blink_timer` (counts **down** with the remaining i-frames/Shield — cosine is
symmetric so a decreasing phase pulses identically):

```python
import math
# phase 0..1 across one INVULN_PULSE_PERIOD-frame cycle
phase = (p.blink_timer % C.INVULN_PULSE_PERIOD) / C.INVULN_PULSE_PERIOD
# cos: phase 0 → +1 → CEIL (bright);  phase 0.5 → −1 → FLOOR (dim);  phase 1 → +1 → CEIL
osc   = 0.5 + 0.5 * math.cos(2 * math.pi * phase)            # 0.0 .. 1.0
alpha = C.INVULN_ALPHA_FLOOR + (C.INVULN_ALPHA_CEIL - C.INVULN_ALPHA_FLOOR) * osc
alpha = int(round(alpha))                                    # 128 .. 255, smoothly
```

- **Floor is respected every frame:** `osc ∈ [0,1]` ⇒ `alpha ∈ [128, 255]`, never below 128, never 0.
  (QA's "ship never fully invisible" check.)
- **Smooth, not 2-state:** `alpha` takes a different value almost every frame across the cycle (e.g. at
  period 30, the down-ramp passes ≈255, 251, 240, 223, 201, 176, … 128 — clearly interpolated).
  (QA's "alpha varies across frames, not a snap" check.)
- **Ends clean:** the existing `if p.invulnerable:` gate is **kept** — when i-frames/Shield reach 0,
  `invulnerable` is False and the ship is drawn **solid (full opacity)** by the normal path. No special
  end-of-pulse handling needed. (QA's "invulnerability still ends → ship returns to solid" check.)

## V11.4 Shield bubble ring during the pulse — STAYS SOLID (the §V2.5 question, answered)

**Decision: the Shield bubble ring does NOT pulse with the ship. It is drawn STEADY at full opacity on
every frame while `shield_active`** — unchanged colour (`BONUS_SHIELD #B4DCFF`), geometry (circle, r=18,
width 2), and centre (player centre). Only the *ship body* pulses.

**Why:** §V2.5's whole job for the ring is to **separate a brief ~1 s i-frame flash from the 5 s Shield** —
"steady blue bubble = I have my shield" vs "no ring = I just got hit." That tell must stay crisp:

- A **steady** ring around a **gently pulsing** ship reads cleanly and keeps the two states distinct (ship
  pulse = "invulnerable right now"; ring = "and it's the 5 s Shield specifically").
- If the ring **faded with the ship**, it would be redundant with the pulse and *weaker* exactly when the
  ship dims — muddying the Shield tell for no benefit.
- **Bonus fix:** under the old code the ring was implicitly skipped on every "off" blink frame (the
  early-`return` bailed before the ring draw), so the ring itself strobed. Drawing it steady every frame
  (it's outside the alpha-surface path — see §V11.5) **removes that strobe too**, making the Shield read
  calmer and more legible than before.

| Element | Behaviour during invuln | Alpha |
|---|---|---|
| Player ship body (triangle + edge + centre dot) | **pulses** 128↔255 on the cosine curve | per §V11.3 |
| Shield bubble ring (`shield_active` only) | **steady**, drawn every frame, no pulse | full (opaque, as today) |

## V11.5 Render mechanism — note for the Programmer (you own the mechanism)

The brief assigns the *render mechanism* to the Programmer; this is orientation, not a lever change.

- `_draw_player` currently draws the ship polygons **straight to `screen`**. A partial alpha needs the
  ship rendered onto a **small temporary surface** first, alpha-modulated, then blitted. Per the brief,
  **size that surface once** (module-level, like the v6 flash surface) — don't allocate per frame. The
  ship's bounding box is `cx-14 … cx+14` × `cy-15 … cy+15` ⇒ **28×30 px** (add ~2 px margin for the edge
  stroke → a 32×34 surface is safe). Draw the ship at local/centre-relative coords on that surface, set
  the alpha, blit so the local centre lands on `(cx, cy)`.
- **Gotcha to avoid a silent no-op:** on a `pygame.SRCALPHA` (per-pixel-alpha) surface, plain
  `surface.set_alpha(a)` is **ignored** — it only affects surface-wide alpha on non-per-pixel surfaces.
  Two reliable ways to apply the §V11.3 `alpha`:
  - **(A) colorkey + `set_alpha`:** make a plain (non-SRCALPHA) surface, fill it with a sentinel colorkey
    colour, `set_colorkey(that)`, draw the ship opaque, then `surf.set_alpha(alpha)` and blit. (The ship
    is non-antialiased solid polygons, so no AA fringe issue.)
  - **(B) SRCALPHA + alpha-channel multiply:** make an `SRCALPHA` surface, draw the ship opaque (alpha
    255), then `surf.fill((255, 255, 255, alpha), special_flags=pygame.BLEND_RGBA_MULT)` to scale the
    alpha channel, then blit.
  Either is fine — Programmer's call. The key is that the chosen path actually honours `alpha` (QA will
  verify the ship is visibly partial, not full or invisible).
- **Keep the cheap common path:** when `not p.invulnerable`, draw straight to `screen` as today (full
  opacity, zero surface cost). Only take the alpha-surface path while invulnerable.
- **Shield ring** (§V11.4) is drawn **straight to `screen` at full alpha every frame** while
  `shield_active` — i.e. *outside* the alpha surface, after the ship blit. Same `pygame.draw.circle`
  call as today, just no longer gated behind the (now-removed) early-return.
- Drive the pulse from `p.blink_timer` (§V11.3) so phase tracks remaining i-frames/Shield. Must not
  regress the smoke gate, the v9 render-smoke (no draw raises), or any AC.
